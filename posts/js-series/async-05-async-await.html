async/await Is Still Promises — Just Prettier

You've used async/await since the JS Click series. You know the syntax. This post is about understanding what's actually happening underneath — because async/await is not a replacement for Promises. It's syntax built on top of Promises. Every async function returns a Promise. Every await unwraps one. The better you understand Promises, the better you write async/await.

🔄

What async Actually Does to a Function

⏸ Pause & Process

async functions always return a Promise — even if you return a plain value. This means calling an async function gives you a Promise back, which you either await or chain with .then(). This is why you can't just do const result = myAsyncFn() and expect result to be the value — it'll be a Promise.

Marking a function async does two things: it allows await to be used inside it, and it wraps the return value in a Promise automatically — even if you return a plain value.

JavaScript — async-returns.js
// async functions ALWAYS return a Promise
async function getValue() {
  return 42  // plain number
}

// But you get a Promise back, not the number
console.log(getValue())  // Promise { 42 }

// To get the value out you need .then() or await
getValue().then(v => console.log(v))  // 42

// Or inside another async function
async function main() {
  const v = await getValue()
  console.log(v)  // 42
}

// Thrown errors become rejected Promises
async function fail() {
  throw new Error("Boom")
}
fail().catch(e => console.log(e.message))  // "Boom"
📍

Where async Functions Can Live

You can put async in front of almost any function form. Knowing all the contexts matters — you'll see them all in real code.

Function Declaration
async function load() { return await fetch(url) }
Arrow Function
const load = async () => { return await fetch(url) }
Object Method
const api = { async getUser() { return await fetch(url) } }
IIFE (Immediately Invoked)
(async () => { const data = await fetch(url) // top-level await workaround })()
ℹ️ Top-Level await

Modern JavaScript modules support top-level await — you can use await outside of any function in a .mjs file or in a module-type script. This makes the IIFE pattern less necessary in modern code. But in classic scripts you still need an async wrapper.

Error Handling Patterns

⏸ Pause & Process

The 'forget await' bug is one of the most common and confusing async mistakes. You call an async function but forget await, and instead of getting the value back, you get a Promise object. Your code tries to use a Promise like a string or number, and everything breaks in ways that look unrelated to the actual mistake. If something looks right but behaves wrong in async code, check for missing await first.

There are three common patterns for handling errors in async/await code. Each has its place.

JavaScript — error-patterns.js
// PATTERN 1: try/catch — most common, cleanest for complex flows
async function loadUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`)
    if (!res.ok) throw new Error(res.status)
    return await res.json()
  } catch (err) {
    console.log("Failed:", err.message)
    return null  // return fallback so caller doesn't crash
  }
}

// PATTERN 2: .catch() on the awaited Promise
// Useful for handling one specific failure inline
async function loadUser2(id) {
  const res = await fetch(`/api/users/${id}`)
    .catch(() => null)  // null on network failure
  if (!res) return null
  return res.json()
}

// PATTERN 3: Go/Error tuple — inspired by Go lang
// Returns [data, null] on success, [null, error] on failure
async function tryCatch(promise) {
  try {
    const data = await promise
    return [data, null]
  } catch (err) {
    return [null, err]
  }
}

// Usage — very clean caller code
const [user, err] = await tryCatch(fetch("/api/user").then(r => r.json()))
if (err) console.log("Error:", err.message)
else console.log("User:", user)
🧠 Mindset Shift

The Go-style tuple pattern is polarizing — some teams love it, some hate it. The point is: there is no single right way to handle async errors. Pick a pattern, apply it consistently across your codebase, and document it. Consistency matters more than which pattern you chose.

The Mistakes Everyone Makes

JavaScript — async-mistakes.js
// ❌ MISTAKE 1: Not awaiting a Promise
async function bad() {
  const data = fetch("/api/user")  // forgot await!
  console.log(data)  // Promise { pending } — not the data
}

// ✅ Always await async operations
async function good() {
  const res = await fetch("/api/user")
  const data = await res.json()
  console.log(data)  // actual data ✓
}

// ❌ MISTAKE 2: Using await in a non-async function
function notAsync() {
  const data = await fetch("/api")  // SyntaxError!
}

// ❌ MISTAKE 3: Expecting await to catch sync errors
// await only catches rejected Promises, not thrown sync errors
async function example() {
  const data = await fetch("/api")  // network errors become rejected Promises ✓
  const parsed = JSON.parse("bad json")  // throws sync — caught by try/catch ✓
  // Both are caught by a wrapping try/catch — but for different reasons
}

// ❌ MISTAKE 4: Async forEach — await inside doesn't work
const ids = [1, 2, 3]
ids.forEach(async (id) => {
  await fetch(`/api/${id}`)  // awaits are NOT awaited by forEach
})
// forEach doesn't wait — all requests fire but you can't know when they finish
// We cover this deeply in Post 06 — it's a big topic

⚡ DEV_FESSION

I once spent an entire afternoon debugging a feature because I'd written const data = fetch("/api/user") — no await. Then I passed data to another function. That function received a pending Promise object, tried to access data.name, got undefined, and everything silently fell apart. No error. Just wrong data everywhere. Adding the missing await fixed all of it. Two characters. Four hours. I now read every async function I write out loud to catch missing awaits before they ruin my afternoon.
