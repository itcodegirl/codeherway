Fetching From Multiple APIs in JavaScript

Combining data from different sources, sequential vs parallel fetching, data transformation, and building a unified layer over messy APIs.

Real Apps Need Data From Multiple Places

Almost every real-world feature pulls from more than one API. A user profile page needs the user record, their posts, and their follower count — potentially from three different endpoints. A product listing needs product data, inventory, and pricing from different services. A weather dashboard needs weather data and location data from different providers.

Managing multiple async data sources cleanly is a core skill. This post covers the patterns you'll use every day: parallel fetching, sequential fetching, data merging, fallback data, and building an abstraction layer so your UI doesn't care where the data comes from.

🔀

Pattern 1: Independent Parallel Fetch

⏸ Pause & Process

The decision between parallel and sequential fetch comes down to one question: does request B need data from request A? If yes, they must be sequential. If no, they should always run in parallel. Running independent requests sequentially is one of the most common (and invisible) performance mistakes in production apps.

When the sources don't depend on each other, run them simultaneously. You've seen this pattern in Post 06 — applying it specifically to multi-API scenarios.

JavaScript — parallel-apis.js
// Three independent APIs — run all at once, merge the results
async function loadDashboard(userId) {
  const [user, stats, notifications] = await Promise.all([
    fetch(`/api/users/${userId}`).then(r => r.json()),
    fetch(`/api/users/${userId}/stats`).then(r => r.json()),
    fetch(`/api/users/${userId}/notifications`).then(r => r.json())
  ])

  return {
    name:          user.name,
    email:         user.email,
    postsCount:    stats.posts,
    followersCount:stats.followers,
    unread:        notifications.filter(n => !n.read).length
  }
}

Pattern 2: Sequential Dependent Fetch

When each source needs data from the previous one. Geolocation → city name → weather is the classic example — you can't skip a step.

JavaScript — sequential-apis.js
// Three APIs in sequence — each depends on the previous result
async function getLocalWeather() {
  // Step 1: Get coordinates from browser geolocation
  const position = await new Promise((resolve, reject) => {
    navigator.geolocation.getCurrentPosition(resolve, reject)
  })
  const { latitude, longitude } = position.coords

  // Step 2: Reverse geocode coordinates → city name
  const geoRes = await fetch(
    `https://nominatim.openstreetmap.org/reverse?lat=${latitude}&lon=${longitude}&format=json`
  )
  const geo      = await geoRes.json()
  const city     = geo.address.city || geo.address.town

  // Step 3: Fetch weather for those coordinates
  const params   = new URLSearchParams({
    latitude, longitude,
    current: "temperature_2m,weather_code,wind_speed_10m",
    temperature_unit: "fahrenheit"
  })
  const weatherRes = await fetch(`https://api.open-meteo.com/v1/forecast?${params}`)
  const weather    = await weatherRes.json()

  return { city, temp: weather.current.temperature_2m }
}

Pattern 3: Parallel With Independent Error Handling

⏸ Pause & Process

The data transformation layer is the professional move that separates 'I fetched data and used it' from 'I fetched data and made it mine.' API responses are shaped for the API's convenience, not yours. Normalizing them into your own consistent format means the rest of your app doesn't need to know how the API works — just how your data looks.

When APIs are independent but some may fail — use Promise.allSettled() and render partial results. Don't let one broken endpoint take down the whole page.

JavaScript — resilient-multi-api.js
// Fetch from multiple sources — show what succeeds, note what fails
async function loadWeatherDashboard(lat, lon) {
  const results = await Promise.allSettled([
    fetchCurrentWeather(lat, lon),
    fetchWeeklyForecast(lat, lon),
    fetchAirQuality(lat, lon)
  ])

  const [current, forecast, airQuality] = results

  return {
    current:    current.status    === "fulfilled" ? current.value    : null,
    forecast:   forecast.status   === "fulfilled" ? forecast.value   : null,
    airQuality: airQuality.status === "fulfilled" ? airQuality.value : null,
    errors: results
      .filter(r => r.status === "rejected")
      .map(r => r.reason.message)
  }
}

// Render — shows what it has, gracefully notes what's missing
const { current, forecast, airQuality, errors } = await loadWeatherDashboard(41.85, -87.65)

if (current)    renderCurrentWeather(current)
if (forecast)   renderForecast(forecast)
if (airQuality) renderAirQuality(airQuality)
if (errors.length) showPartialErrorBanner(errors)

Pattern 4: The Data Transformation Layer

Different APIs return data in different shapes. Normalizing it into a consistent format before it reaches your UI means the UI only has to understand one shape — yours.

JavaScript — data-transform.js
// Raw API responses have different shapes — normalize them

// Open-Meteo returns this:
// { current: { temperature_2m: 72, wind_speed_10m: 12, weather_code: 3 } }

// Nominatim returns this:
// { address: { city: "Chicago", state: "Illinois", country: "United States" } }

// Your UI wants this — one clean object:
function normalizeWeatherData(weatherRaw, locationRaw) {
  const weather  = weatherRaw.current
  const address  = locationRaw.address

  return {
    city:        address.city || address.town || "Unknown",
    region:      address.state,
    country:     address.country,
    tempF:       Math.round(weather.temperature_2m),
    windMph:     Math.round(weather.wind_speed_10m),
    condition:   getConditionLabel(weather.weather_code),
    fetchedAt:   new Date().toISOString()
  }
}

// Now the UI just uses .city, .tempF, .condition — never touches raw API shapes
function getConditionLabel(code) {
  if (code === 0)            return "Clear"
  if (code <= 3)            return "Partly cloudy"
  if (code >= 61 && code <= 67) return "Rain"
  if (code >= 71 && code <= 77) return "Snow"
  if (code >= 95)            return "Thunderstorm"
  return "Cloudy"
}
🧠 Mindset Shift

The transformation layer is a separation of concerns. Your data layer speaks API. Your UI layer speaks your app's internal format. When an API changes its response shape, you update one normalization function — not every place in your UI that touches that data. This is how real codebases stay maintainable as they grow.

⚡ DEV_FESSION

I once integrated a third-party API directly into my UI — no normalization layer. Their field was temp_celsius. I referenced it in twelve places. Then they renamed it to temperature_c in a breaking version update. I had to find and fix twelve spots. If I'd written a normalize function and referenced weatherData.temp everywhere, I'd have fixed one line. The API changed once. My suffering was twelve times more than it needed to be.
