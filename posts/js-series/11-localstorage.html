JavaScript localStorage & sessionStorage

Persisting data between page loads, caching API responses, saving user preferences — without a database.

What They Are and Why They Matter

localStorage and sessionStorage are key-value stores built into the browser. They let you save data on the client — no server, no database, no network request required. Close a tab, reopen it, and localStorage is still there. sessionStorage clears when the tab closes.

These come up constantly in real projects and portfolio work: caching API responses, remembering a user's last search, persisting a shopping cart, saving theme preferences. And they're a staple of frontend interviews. You need to know them cold.

⚖️

localStorage vs sessionStorage

⏸ Pause & Process

Everything in localStorage is a string. That's the rule and the source of the most common localStorage bugs. localStorage.setItem('count', 5) stores the string "5", not the number 5. Always use JSON.stringify() when storing and JSON.parse() when reading back. Wrap the parse in try/catch — if the stored value is corrupt, JSON.parse() will throw.

localStorage
Persists until explicitly cleared. Survives tab closes, browser restarts. Shared across all tabs on the same origin. Use for: preferences, cached data, saved state.
sessionStorage
Lives only for the current tab session. Clears on tab close. Not shared between tabs. Use for: temporary wizard state, form progress, one-time flows.
🔧

The Core API

JavaScript — storage-api.js
// The same 4 methods work on both localStorage and sessionStorage

// SET — values must be strings
localStorage.setItem("theme", "dark")
localStorage.setItem("count", String(42))          // numbers → strings
localStorage.setItem("user", JSON.stringify({ name: "Jenna" }))  // objects → JSON

// GET — always returns a string or null
localStorage.getItem("theme")       // "dark"
localStorage.getItem("missing")     // null — key doesn't exist

// Parse back to original type
const count = Number(localStorage.getItem("count"))
const user  = JSON.parse(localStorage.getItem("user"))

// REMOVE — delete a single key
localStorage.removeItem("theme")

// CLEAR — delete everything (use with caution)
localStorage.clear()

// CHECK LENGTH — how many keys are stored
console.log(localStorage.length)

Safe Storage Wrapper

⏸ Pause & Process

The TTL (time-to-live) cache pattern is genuinely useful for real projects: store data along with a timestamp, check on retrieval whether the data has expired, re-fetch if it has. This gives you the performance benefit of caching without serving users stale data indefinitely. It's the pattern the Capstone uses.

Raw localStorage is fine for simple cases, but it has footguns: it throws if storage is full, it can throw in private browsing mode in some browsers, and parsing JSON without error handling will crash on corrupted data. A wrapper handles all of that.

JavaScript — storage-wrapper.js
// A safe storage wrapper — handles JSON, errors, and type coercion
const storage = {
  get(key, fallback = null) {
    try {
      const item = localStorage.getItem(key)
      return item !== null ? JSON.parse(item) : fallback
    } catch {
      return fallback  // corrupted JSON or storage unavailable
    }
  },

  set(key, value) {
    try {
      localStorage.setItem(key, JSON.stringify(value))
      return true
    } catch {
      console.warn("Storage write failed", key)
      return false  // storage full or unavailable
    }
  },

  remove(key) {
    try { localStorage.removeItem(key) } catch {}
  },

  has(key) {
    return localStorage.getItem(key) !== null
  }
}

// Clean usage — no JSON.parse/stringify clutter at the call site
storage.set("user", { name: "Jenna", theme: "dark" })
const user = storage.get("user", {})  // {} is the fallback
console.log(user.name)  // "Jenna"

Caching API Responses

One of the most practical uses: cache API data to avoid redundant network requests. The pattern: check localStorage first, return cached data if it exists and isn't too old, fetch and cache if not.

JavaScript — api-cache.js
// Cache API responses with a TTL (time to live)
async function fetchWithCache(url, ttlMs = 5 * 60 * 1000) {
  const cacheKey = `cache:${url}`
  const cached   = storage.get(cacheKey)

  if (cached && Date.now() - cached.timestamp < ttlMs) {
    console.log("Cache hit:", url)
    return cached.data  // fresh cache — no network request
  }

  console.log("Cache miss — fetching:", url)
  const res  = await fetch(url)
  const data = await res.json()

  storage.set(cacheKey, { data, timestamp: Date.now() })
  return data
}

// Second call returns cached data — no network trip
await fetchWithCache("https://jsonplaceholder.typicode.com/users")
await fetchWithCache("https://jsonplaceholder.typicode.com/users")  // cache hit
⚡ This Is Your Capstone Cache

The Post 18 weather dashboard uses this exact caching pattern to store the last fetched weather data. When a user revisits the page, they see cached data instantly while a fresh fetch happens in the background. This is called stale-while-revalidate and it makes UIs feel dramatically faster.

What NOT to Store in localStorage

⚠️ Security Warning

localStorage is accessible by any JavaScript running on your page — including third-party scripts. Never store passwords, authentication tokens, credit card numbers, or any truly sensitive user data. localStorage is vulnerable to XSS attacks. Authentication tokens belong in httpOnly cookies managed by your server — not in localStorage.

📡

The storage Event

When localStorage changes in one tab, other tabs on the same origin receive a storage event. This is the native way to sync state across tabs without a backend.

JavaScript — storage-event.js
// Listen for changes from OTHER tabs (not the current one)
window.addEventListener("storage", (e) => {
  console.log("Key changed:", e.key)
  console.log("Old value:", e.oldValue)
  console.log("New value:", e.newValue)

  if (e.key === "theme") {
    applyTheme(e.newValue)  // sync theme across tabs instantly
  }
})

// Example: user changes theme in Tab A → Tab B updates automatically
// This fires ONLY in other tabs — not the tab that made the change
⚡ DEV_FESSION

I built a dark mode toggle for a project and was so proud of it. Then I realized refreshing the page reset it to light mode every time because I never persisted the preference. Fixing it took two lines: localStorage.setItem("theme", mode) on toggle and storage.get("theme", "light") on page load. Two lines. I had shipped something that forgot what you told it the moment you refreshed. localStorage is the difference between an app that remembers you and one that makes you repeat yourself constantly.
