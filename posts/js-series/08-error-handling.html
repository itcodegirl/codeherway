Async Error Handling in JavaScript

Retry logic, exponential backoff, custom error classes, user-facing error states, and the art of failing gracefully.

Errors Will Happen. The Question Is How You Handle Them.

Every fetch call you make can fail. Networks are unreliable. Servers go down. APIs rate-limit you. Users have bad connections. The difference between amateur and professional async code isn't whether errors are handled — it's how they're handled. A senior developer thinks about failure modes before writing the happy path.

This post builds on the basic try/catch you already know and goes deeper: the types of async errors, custom error classes, retry logic, exponential backoff, and showing the user something useful when things go wrong.

🗂️

The Four Types of Async Errors

⏸ Pause & Process

Custom error classes are one of those things that feels like over-engineering until you're debugging a production issue at midnight and you can't tell a network error from a validation error from an auth error. Giving each error type its own class with its own properties means you can catch them specifically and respond correctly — not just log a generic message and hope.

Not all async errors are the same. Identifying which type you're dealing with tells you how to respond.

Network
Network / Connection Error
No internet, DNS failure, CORS block. fetch() throws an actual Error — the Promise rejects. response.ok is never reached.
HTTP
HTTP Error Status
Server responded — but with 4xx or 5xx. fetch() doesn't throw. response.ok is false. You must throw manually.
Parse
JSON Parse Error
Server responded with 200 but the body isn't valid JSON. response.json() throws. Rare but worth handling.
App
Application Logic Error
Data came back but missing expected fields. Your code throws because it tried to access something that doesn't exist.
🏷️

Custom Error Classes

JavaScript's built-in Error is fine, but in a real app you want to distinguish between error types in your catch blocks. Custom error classes let you do that cleanly.

JavaScript — custom-errors.js
// Custom error classes — extend the built-in Error
class NetworkError extends Error {
  constructor(message) {
    super(message)
    this.name = "NetworkError"
  }
}

class HttpError extends Error {
  constructor(status, message) {
    super(message)
    this.name   = "HttpError"
    this.status = status
  }
}

// Use them in your fetch wrapper
async function apiFetch(url, options = {}) {
  let res
  try {
    res = await fetch(url, options)
  } catch {
    throw new NetworkError("No connection — check your internet")
  }
  if (!res.ok) throw new HttpError(res.status, res.statusText)
  return res.json()
}

// Now catch blocks can respond intelligently by type
try {
  const data = await apiFetch("/api/user")
} catch (err) {
  if (err instanceof NetworkError) {
    showToast("No internet connection")
  } else if (err instanceof HttpError && err.status === 401) {
    redirectToLogin()
  } else if (err instanceof HttpError && err.status >= 500) {
    showToast("Server is having issues — try again later")
  } else {
    showToast("Something went wrong")
  }
}

Retry Logic

⏸ Pause & Process

Exponential backoff sounds scary. It's just: wait a bit, try again. Wait a bit longer, try again. Wait even longer, try again. The 'exponential' part means each retry doubles the wait time — so you're not hammering a struggling server. Most real retry implementations cap it at 3-5 attempts with a maximum delay. That's it.

Some errors are transient — a brief network blip, a server momentarily overloaded. Retrying once or twice often succeeds without the user even knowing there was a problem.

JavaScript — retry.js
// Retry a Promise-returning function up to maxRetries times
async function withRetry(fn, maxRetries = 3, delayMs = 1000) {
  let lastError

  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      return await fn()  // success — return immediately
    } catch (err) {
      lastError = err
      console.log(`Attempt ${attempt} failed. Retrying in ${delayMs}ms...`)

      // Don't wait after the last attempt
      if (attempt < maxRetries) {
        await new Promise(r => setTimeout(r, delayMs))
      }
    }
  }

  throw lastError  // all retries exhausted — let the caller handle it
}

// Usage
const data = await withRetry(
  () => fetch("/api/user").then(r => r.json()),
  3,    // try up to 3 times
  500   // wait 500ms between retries
)
📈

Exponential Backoff

Fixed-delay retries can overwhelm a struggling server — every client retrying at the same interval creates synchronized traffic spikes. Exponential backoff solves this by doubling the wait time after each failure, plus a small random amount (called jitter) to spread out the traffic.

JavaScript — exponential-backoff.js
async function withExponentialBackoff(fn, maxRetries = 4) {
  let lastError

  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      return await fn()
    } catch (err) {
      lastError = err

      if (attempt < maxRetries - 1) {
        // 2^attempt * 1000ms + up to 1000ms random jitter
        // attempt 0: ~1000ms, attempt 1: ~2000ms, attempt 2: ~4000ms
        const delay = (2 ** attempt * 1000) + (Math.random() * 1000)
        await new Promise(r => setTimeout(r, delay))
      }
    }
  }

  throw lastError
}

// Only retry on server errors — not on client errors (400, 401, 404)
async function smartRetry(fn) {
  return withExponentialBackoff(async () => {
    try {
      return await fn()
    } catch (err) {
      // Don't retry client errors — they won't magically fix themselves
      if (err instanceof HttpError && err.status < 500) throw err
      throw err  // server errors — backoff will retry
    }
  })
}
🧠 Mindset Shift

Don't retry 4xx errors — they're your fault (bad request, unauthorized, not found). A 401 isn't going to fix itself with more attempts. Retry 5xx errors and network errors — those are server-side and transient. Smart retry logic knows the difference.

User-Facing Error States

Technical errors need to be translated into human language. Users don't care about HTTP status codes. They care whether they can continue doing what they were trying to do.

JavaScript — user-error-states.js
// Translate technical errors into user messages
function getUserMessage(err) {
  if (err instanceof NetworkError) {
    return "Can't connect. Check your internet and try again."
  }
  if (err instanceof HttpError) {
    if (err.status === 401) return "You need to log in to do that."
    if (err.status === 403) return "You don't have permission for this."
    if (err.status === 404) return "That resource doesn't exist anymore."
    if (err.status === 429) return "Too many requests. Wait a moment and try again."
    if (err.status >= 500) return "Something went wrong on our end. Try again shortly."
  }
  return "Something unexpected happened. Please try again."
}

// Pattern: loading → success → error states in the UI
async function loadAndRender() {
  const container = document.querySelector("#content")
  container.innerHTML = "<p>Loading...</p>"

  try {
    const data = await apiFetch("/api/posts")
    container.innerHTML = data.map(p => `<p>${p.title}</p>`).join("")
  } catch (err) {
    container.innerHTML = `
      <div class="error-state">
        <p>${getUserMessage(err)}</p>
        <button onclick="loadAndRender()">Try again</button>
      </div>
    `
  }
}
💡 Pro Tip

Always give users a way to recover. An error message with a "Try again" button is infinitely better than a blank screen. The button costs you two lines of code and saves your users from thinking your app is broken.

⚡ DEV_FESSION

I shipped a feature where any failed API call just… left the page blank. No error message. No retry button. Just nothing where content should be. Users assumed the feature didn't exist. Support got tickets. My manager got questions. I spent two hours in a meeting explaining why the page was blank instead of showing an error. Writing container.innerHTML = "Something went wrong. Try again." would have taken me 30 seconds. I have never shipped a blank error state since.
