Pattern 5: Stale-While-Revalidate

The name is from an HTTP cache directive, but the pattern shows up everywhere in JavaScript land. The idea: when someone requests data, give them the old cached version immediately, then quietly fetch fresh data in the background and update the UI when it arrives.

The user always sees something instantly. No spinner, no flash of empty. It might be slightly stale for half a second, but for most UI data (user profile, news feed, dashboard stats), that's completely fine.

❌ Cache-then-network (sequential)
// Check cache first, // THEN go to network async function getUser(id) { const cached = getCache(id); if (cached) return cached; // Only hits network // if cache miss const fresh = await fetchUser(id); setCache(id, fresh); return fresh; } // Problem: cached data gets stale // and never refreshes
✅ Stale-while-revalidate
// Return stale + refresh both function getUser(id, onUpdate) { const cached = getCache(id); // Show stale data instantly if (cached) onUpdate(cached); // Always revalidate in BG fetchUser(id).then(fresh => { setCache(id, fresh); onUpdate(fresh); // update UI }); } // Instant response + always fresh

Full SWR Implementation

swr.js
const swrCache = new Map(); function useSWR(key, fetcher, { maxAge = 60000 } = {}) { const now = Date.now(); const callbacks = new Set(); function subscribe(cb) { callbacks.add(cb); return () => callbacks.delete(cb); } function notify(data, isStale = false) { callbacks.forEach(cb => cb({ data, isStale })); } async function revalidate() { try { const fresh = await fetcher(key); swrCache.set(key, { data: fresh, timestamp: Date.now() }); notify(fresh, false); } catch (err) { console.error('SWR revalidation failed:', err); } } const cached = swrCache.get(key); const isExpired = cached && (now - cached.timestamp > maxAge); if (cached) { // Immediately emit stale data setTimeout(() => notify(cached.data, isExpired), 0); } // Always revalidate if no cache, or if expired if (!cached || isExpired) { revalidate(); } return { subscribe, revalidate }; } // Usage const userSWR = useSWR( 'user-123', (id) => fetch(`/api/users/${id}`).then(r => r.json()), { maxAge: 30000 } // 30 second max age ); userSWR.subscribe(({ data, isStale }) => { renderUser(data); // Optionally show a subtle "refreshing" indicator if (isStale) showRefreshingBadge(); });
✅ Real Talk

Libraries like SWR and React Query are entire packages built around this pattern. Understanding the pattern helps you use those libraries well — and makes you dangerous in interviews when you can explain what they're actually doing under the hood.

Pattern 6: Infinite Scroll

Load more content as the user scrolls. Sounds simple. Gets messy fast. The key tool here is IntersectionObserver — a browser API that fires when an element enters or leaves the viewport, without any scroll event listeners (which are notoriously bad for performance).

How Infinite Scroll Works

1
Place a sentinel element

A small invisible div at the bottom of your list. This is what we observe.

2
Watch it with IntersectionObserver

When it enters the viewport, trigger a fetch for the next page.

3
Append new items

New content pushes the sentinel down. User scrolls more. Cycle continues.

4
Stop observing when done

When the API says there's no next page, disconnect the observer.

infinite-scroll.js
let page = 1; let isLoading = false; let hasMore = true; const list = document.getElementById('posts-list'); const sentinel = document.getElementById('scroll-sentinel'); async function loadNextPage() { // Guard against overlapping requests if (isLoading || !hasMore) return; isLoading = true; // Show skeleton loaders at the bottom const skeletons = renderSkeletons(3); list.appendChild(skeletons); try { const response = await fetch(`/api/posts?page=${page}&limit=10`); const { posts, nextPage } = await response.json(); // Remove skeletons, add real content skeletons.remove(); posts.forEach(post => { list.insertAdjacentHTML('beforeend', renderPost(post)); }); // Update state page = nextPage || page + 1; hasMore = !!nextPage; if (!hasMore) { sentinel.remove(); // No more content, stop observing list.insertAdjacentHTML('beforeend', '<p class="end-msg">You\'ve seen it all. Impressive.</p>'); } } catch (err) { skeletons.remove(); showError('Failed to load more posts'); } finally { isLoading = false; } } // The observer — no scroll listeners needed const observer = new IntersectionObserver( (entries) => { if (entries[0].isIntersecting) { loadNextPage(); } }, { rootMargin: '200px' } // Trigger 200px before sentinel is visible ); observer.observe(sentinel); loadNextPage(); // Load first page immediately
⚠️ Infinite Scroll Gotchas

Accessibility: Users who keyboard-navigate can't reach content that hasn't loaded. Consider a "Load More" button as a fallback.
Back button: If the user clicks into a post and returns, they lose their scroll position. This is why many apps use cursor-based pagination with URL state.
Memory: A list with 10,000 DOM nodes will destroy performance. For very long lists, use virtual scrolling.

Pattern 7: Prefetching

Prefetching is fetching data before the user explicitly asks for it, based on what they're probably about to do. Hover over a link? Start fetching that page's data. Scrolling through a list? Fetch the next page before they reach the bottom.

Used well, prefetching makes your app feel psychic. Used wrong, it hammers your API for data users never actually view.

Hover Prefetch

hover-prefetch.js
const prefetchCache = new Map(); async function prefetch(url) { // Already cached or in-flight? Skip if (prefetchCache.has(url)) return; // Store the promise so we don't double-fetch const promise = fetch(url).then(r => r.json()); prefetchCache.set(url, promise); } async function getFromCache(url) { // If we prefetched this, get it instantly if (prefetchCache.has(url)) { return await prefetchCache.get(url); } // Otherwise fetch normally const data = await fetch(url).then(r => r.json()); return data; } // Attach to links — prefetch on hover document.querySelectorAll('a[data-prefetch]').forEach(link => { let timer; link.addEventListener('mouseenter', () => { // Wait 100ms to avoid prefetching on accidental hovers timer = setTimeout(() => { const apiUrl = link.getAttribute('data-api-url'); if (apiUrl) prefetch(apiUrl); }, 100); }); link.addEventListener('mouseleave', () => { clearTimeout(timer); }); link.addEventListener('click', async (e) => { e.preventDefault(); const apiUrl = link.getAttribute('data-api-url'); const data = await getFromCache(apiUrl); renderPage(data); // Probably already in cache = instant }); });

Viewport-Based Prefetch (Scroll Ahead)

prefetch-next-page.js
// When user is 70% through the current page, // start fetching the next one let nextPagePrefetched = null; window.addEventListener('scroll', debounce(() => { const scrollPercent = (window.scrollY + window.innerHeight) / document.body.scrollHeight; if (scrollPercent > 0.7 && !nextPagePrefetched && hasMore) { nextPagePrefetched = fetch(`/api/posts?page=${page + 1}`).then(r => r.json()); console.log('Prefetching next page...'); } }, 100)); async function loadNextPage() { // If we prefetched, this resolves instantly const data = nextPagePrefetched ? await nextPagePrefetched : await fetch(`/api/posts?page=${page}`).then(r => r.json()); nextPagePrefetched = null; // Reset for next page renderPosts(data.posts); }

Pattern 8: Background Sync

Background sync is what happens when a user performs an action while offline (or on a flaky connection), and the app queues that action to send when connectivity is restored. Done right, the user doesn't even know they were offline.

The full-power version involves Service Workers and the Background Sync API. Here's the vanilla JS approximation that works for most use cases:

sync-queue.js
// A queue of actions to sync when online class SyncQueue { constructor() { // Restore any pending actions from last session this.queue = JSON.parse(localStorage.getItem('sync-queue') || '[]'); this._bindListeners(); } add(action) { this.queue.push({ ...action, id: Date.now().toString(), timestamp: new Date().toISOString() }); this._persist(); // If online, sync immediately if (navigator.onLine) this.flush(); } async flush() { if (this.queue.length === 0) return; console.log(`Syncing ${this.queue.length} pending action(s)...`); const toProcess = [...this.queue]; for (const action of toProcess) { try { await this._execute(action); // Remove from queue on success this.queue = this.queue.filter(a => a.id !== action.id); this._persist(); } catch (err) { console.error('Action failed, will retry:', action.type, err.message); break; // Stop processing on failure (maintain order) } } } async _execute(action) { switch (action.type) { case 'CREATE_POST': return fetch('/api/posts', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify(action.payload) }); case 'UPDATE_PROFILE': return fetch('/api/profile', { method: 'PATCH', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify(action.payload) }); default: throw new Error(`Unknown action type: ${action.type}`); } } _persist() { localStorage.setItem('sync-queue', JSON.stringify(this.queue)); } _bindListeners() { // Flush when connection is restored window.addEventListener('online', () => { console.log('Back online! Syncing...'); this.flush(); }); } } // Usage const syncQueue = new SyncQueue(); // User posts a comment — add to queue function submitComment(text) { // Optimistic update (from Post 15) addCommentToUI(text); // Queue the actual API call syncQueue.add({ type: 'CREATE_POST', payload: { text, authorId: currentUser.id } }); }
⚡ Online / Offline Detection

navigator.onLine tells you the current state. window.addEventListener('online', ...) and window.addEventListener('offline', ...) fire when it changes. These are your tools for offline-aware UIs.

SWR: Show stale, revalidate in background. Always instant, always eventually fresh.
