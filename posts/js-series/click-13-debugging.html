Errors Are Not Failures. They're Feedback.

Every developer encounters errors constantly — including seniors. The difference is that experienced developers know how to read them, find them fast, and fix them without panic. This post turns you into that developer.

🚨

The Four Error Types You'll See Most

⏸ Pause & Process

The console is more than just console.log(). console.table() displays arrays and objects in a formatted table — genuinely useful when debugging a list of user data. console.group() lets you collapse related logs. console.time() measures performance. You don't need all of these, but knowing console.table() alone will save you time every week.

SyntaxError
const x = { ← missing closing brace
Invalid JavaScript syntax. Code won't run at all.
ReferenceError
console.log(username) ← never declared
Variable doesn't exist in this scope.
TypeError
Cannot read properties of null (reading 'classList')
The #1 beginner error. You tried to use a method on null/undefined — usually because querySelector found nothing.
RangeError
Maximum call stack size exceeded
Value out of valid range — often infinite recursion.
ℹ️ Reading a TypeError

The most common error beginners see: Cannot read properties of null (reading 'classList'). This means querySelector returned null (element not found) and you tried to use .classList on it. The fix: check your selector, make sure the element exists in HTML, and ensure your script runs after the DOM loads.

Console Debugging: Your First Line of Defense

⏸ Pause & Process

try/catch has one critical gotcha: it only catches synchronous errors. If the error happens inside a Promise or async function, try/catch in the outer scope won't catch it. We cover async error handling properly in the next post — and it gets its own deep dive in the Async series. For now: try/catch is your tool for synchronous code and JSON parsing.

💡 DevTools Breakpoints

In Chrome: DevTools → Sources tab. In Firefox: DevTools → Debugger tab. Click any line number to set a breakpoint — execution pauses there and you can inspect every variable's value at that exact moment. Far more powerful than console.log for complex bugs.

try / catch: Handle Errors Gracefully

JavaScript — try-catch.js
// Without try/catch — one bad line crashes everything after it JSON.parse("this is not json") // throws SyntaxError — script stops console.log("never runs") // With try/catch — error is caught, execution continues try { const data = JSON.parse("invalid json") } catch (error) { console.error(error.name) // "SyntaxError" — the error type console.error(error.message) // "Unexpected token i in JSON..." } finally { // runs whether try succeeded or catch fired // use this to hide loading spinners, re-enable buttons, clean up hideLoadingSpinner() } // Throw your own errors with meaningful messages function divide(a, b) { if (b === 0) throw new Error("Division by zero") return a / b } try { divide(10, 0) } catch (e) { console.log(e.name) // "Error" console.log(e.message) // "Division by zero" }
🧠 When to Use try/catch

Use try/catch around code that can fail for reasons outside your control: JSON.parse(), network requests, anything that depends on external data. Don't wrap every line — only the parts that might throw. Use finally to restore UI state (hide spinners, re-enable buttons) regardless of whether an error occurred.

⚡ DEV_FESSION

My first live bug was a TypeError that crashed the whole page because I forgot a null check. A user clicked a button, querySelector found nothing, and .classList.add() threw. Nothing on the page worked after that. Now I defensively check every DOM query and wrap anything async in try/catch. One caught error is worth a hundred logged ones.

🏆MINI BUILD — Break It Intentionally, Then Fix It

The best way to get comfortable with errors is to cause them on purpose.

Your steps:
  • 01 Try console.log(x) before declaring x with let — see the TDZ error. Compare with var — see undefined.
  • 02 Call null.classList.add("x") — read the TypeError message carefully: "Cannot read properties of null (reading 'classList')"
  • 03 Wrap JSON.parse("bad json") in try/catch — log error.name and error.message separately
  • 04 Add a finally block that always runs — log "cleanup complete" to confirm it fires both on success and error
  • 05 Write a function that throws a custom new Error("Your message") and catch it
  • 06 Bonus: Use DevTools breakpoints (Sources/Debugger tab) to pause execution and inspect variable values
ℹ️ Async Errors Are Different

The try/catch you just learned handles synchronous errors. Async code — fetch, Promises, async/await — has its own error patterns. We cover those in Post 14: Async JavaScript, and the full deep-dive is in the Async series.
