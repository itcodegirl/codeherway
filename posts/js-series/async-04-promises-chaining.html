Why .then() Returns a Promise

In Post 03 you learned the basic Promise API. Now we go deeper. The most important thing to understand about .then() is that it always returns a new Promise. That's what makes chaining possible — and it's the core mechanic that replaced callback hell with clean, flat, readable code.

When you return a value from inside .then(), that value becomes the resolved value of the next Promise in the chain. When you return another Promise, the chain waits for it to settle before continuing.

🔗

Chaining .then()

⏸ Pause & Process

The key to understanding chaining is this: .then() always returns a new Promise. Whatever you return from inside .then() becomes the resolved value of that new Promise. Return nothing → undefined gets passed down. Return a value → that value gets passed down. Return another Promise → the chain waits for it. Once you internalize this, chains stop feeling like magic.

JavaScript — chaining.js
// Each .then() receives the return value of the previous one
Promise.resolve(1)
  .then(n => n + 1)       // 2
  .then(n => n * 10)      // 20
  .then(n => console.log(n))  // 20

// Real-world chain — fetch → parse → transform → use
fetch("https://jsonplaceholder.typicode.com/users/1")
  .then(response => {
    if (!response.ok) throw new Error(response.status)
    return response.json()          // returns a Promise — chain waits
  })
  .then(user => {
    return { name: user.name, city: user.address.city }  // transform
  })
  .then(simplified => console.log(simplified))
  .catch(err => console.log("Failed:", err.message))
🧠 Mindset Shift

Think of a Promise chain as a pipeline. Data enters at the top and flows through each .then() stage, getting transformed at each step. Each stage only runs after the previous one completes. One .catch() at the bottom catches any error from any stage above it — you don't repeat error handling at every level. That was the whole point.

🛑

Error Propagation in Chains

When an error is thrown (or a Promise rejects) anywhere in the chain, it skips all remaining .then() handlers and jumps straight to the nearest .catch(). This is the power of centralized error handling.

JavaScript — error-propagation.js
Promise.resolve("start")
  .then(val => {
    console.log("Step 1:", val)   // runs ✓
    throw new Error("Something broke")
  })
  .then(val => {
    console.log("Step 2:", val)   // SKIPPED — error in step 1
  })
  .then(val => {
    console.log("Step 3:", val)   // SKIPPED
  })
  .catch(err => {
    console.log("Caught:", err.message)  // "Caught: Something broke" ✓
  })
  .then(() => {
    console.log("After catch — chain continues") // runs ✓
  })
💡 Pro Tip

You can recover from errors mid-chain. If your .catch() returns a value instead of re-throwing, the chain continues from there as if nothing happened. This lets you provide fallback data: .catch(() => defaultUser) — the next .then() gets defaultUser and keeps going.

Promise.all — Run in Parallel, Wait for All

⏸ Pause & Process

Promise.all() vs Promise.allSettled() is a real architectural decision. Use Promise.all() when all requests must succeed (one failure means the whole operation failed). Use Promise.allSettled() when you want all results regardless — you can handle each success and failure individually. For most UI dashboards, allSettled() gives a better user experience.

The most important static composition method. Pass an array of Promises, get back a single Promise that resolves when all of them resolve — with an array of all their values in order. If any one rejects, the whole thing rejects immediately.

JavaScript — promise-all.js
// Three fetches running in parallel
const userReq  = fetch("https://jsonplaceholder.typicode.com/users/1")
const postReq  = fetch("https://jsonplaceholder.typicode.com/posts/1")
const albumReq = fetch("https://jsonplaceholder.typicode.com/albums/1")

Promise.all([userReq, postReq, albumReq])
  .then(([userRes, postRes, albumRes]) => {
    // All three destructured in order — all Responses
    return Promise.all([
      userRes.json(),
      postRes.json(),
      albumRes.json()
    ])
  })
  .then(([user, post, album]) => {
    console.log(user.name, post.title, album.title)
  })
  .catch(err => console.log("One failed:", err))

// Total time = slowest request, not sum of all requests
// Sequential would take: req1 + req2 + req3
// Parallel takes:        max(req1, req2, req3)  ← much faster
🗂️

The Full Static Method Toolkit

Promise.all()
Use when: all must succeed
Resolves when ALL Promises fulfill. Rejects immediately if ANY rejects. Returns array of all values.
Promise.allSettled()
Use when: some may fail
Waits for ALL to settle (fulfilled or rejected). Never rejects. Returns array of result objects with status.
Promise.race()
Use when: fastest wins
Resolves or rejects as soon as the FIRST Promise settles. Used for timeouts and fallbacks.
Promise.any()
Use when: one success is enough
Resolves with the first FULFILLED value. Only rejects if ALL reject. Ignores individual rejections.
JavaScript — composition-methods.js
// Promise.allSettled — when some might fail, you still want all results
const requests = [
  fetch("/api/users"),
  fetch("/api/broken-endpoint"),
  fetch("/api/posts")
]

Promise.allSettled(requests).then(results => {
  results.forEach((result, i) => {
    if (result.status === "fulfilled") {
      console.log(`Request ${i} succeeded`)
    } else {
      console.log(`Request ${i} failed:`, result.reason)
    }
  })
})

// Promise.race — add a timeout to any async operation
function withTimeout(promise, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error(`Timed out after ${ms}ms`)), ms)
  )
  return Promise.race([promise, timeout])
}

withTimeout(fetch("/api/slow-endpoint"), 3000)
  .then(res => res.json())
  .catch(err => console.log(err.message))  // "Timed out after 3000ms"
ℹ️ allSettled vs all — Know Which to Reach For

Use Promise.all() when every request must succeed for the feature to work — like loading required data before rendering a page. Use Promise.allSettled() when you want to show partial results even if some fail — like a dashboard where widgets load independently and a broken widget shouldn't hide the others.

⚡ DEV_FESSION

I used Promise.all() for a dashboard that loaded four independent widgets. One widget's endpoint was flaky and would occasionally fail — and when it did, it took the whole dashboard down with it. Every widget. Gone. I switched to Promise.allSettled() in about two minutes and suddenly flaky widgets just showed an error state while the other three loaded perfectly. Knowing which tool to reach for is half the job.
