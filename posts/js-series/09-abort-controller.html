JavaScript AbortController

Stale searches, navigation away, double-submits — here's how to cancel in-flight requests before they cause problems.

The Problem: Requests Don't Know When to Stop

Here's a scenario every developer hits eventually: a user types into a search box, triggering a fetch. They keep typing. Three more fetches fire. The first one — for an old search term — responds last and overwrites the results from the correct current search. The UI shows the wrong data. No error. No warning. Just stale results confidently displayed as if they're correct.

Or: a user navigates away from a page. The fetch for that page is still in-flight. It resolves, tries to update state, and throws because the component is gone. AbortController is the native solution to both problems.

🧩

How AbortController Works

⏸ Pause & Process

The pattern is always the same: create a controller, pass its signal to fetch, call abort() when you want to cancel. The key insight is that you need to create a new AbortController for each request (or set of related requests). Reusing a controller after aborting it won't work — once aborted, always aborted.

1
Create an AbortController instance — it has a signal property
2
Pass signal to fetch() in the options object
3
When you want to cancel, call controller.abort()
4
The fetch rejects with an AbortError — catch it and ignore it
JavaScript — abort-basics.js
// The minimal AbortController pattern
const controller = new AbortController()
const { signal } = controller

// Pass signal to fetch
fetch("/api/search?q=hello", { signal })
  .then(res => res.json())
  .then(data => console.log(data))
  .catch(err => {
    if (err.name === "AbortError") {
      console.log("Request cancelled — no problem")
      return  // not a real error — just skip it
    }
    console.log("Real error:", err)
  })

// Cancel after 2 seconds
setTimeout(() => controller.abort(), 2000)

The Search Input Pattern

The most common real-world use case: cancel the previous search request every time the user types a new character. Only the most recent request matters.

JavaScript — search-cancel.js
let currentController = null

async function search(query) {
  // Cancel any in-flight request from the previous keystroke
  if (currentController) {
    currentController.abort()
  }

  // Create a fresh controller for this request
  currentController = new AbortController()

  try {
    const res = await fetch(
      `/api/search?q=${encodeURIComponent(query)}`,
      { signal: currentController.signal }
    )
    const data = await res.json()
    renderResults(data)
  } catch (err) {
    if (err.name === "AbortError") return  // stale — ignore
    showError(err)
  }
}

// Wire up the input
document.querySelector("#search").addEventListener("input", e => {
  search(e.target.value)
})
⚡ AbortError Is Not a Real Error

When you abort a request, it throws an error with name === "AbortError". This is not a problem — it's expected behavior. Always check for it first in your catch block and return early. Passing it to your error handler would show the user a fake error for something that worked perfectly.

💡 Even Better With Debounce

The search input pattern above cancels the previous request before firing a new one. But you can go further: combine it with debounce so the fetch doesn't even trigger until the user stops typing. That combo — debounce + AbortController — is the production-grade version. We build it in Post 13: Debounce & Throttle.

Cleanup on Navigation

⏸ Pause & Process

Timeout handling with AbortController is something developers often reinvent badly. The clean way: create a controller, set a setTimeout that calls controller.abort(), pass the signal to fetch. If the fetch completes in time, clear the timeout. If not, the abort fires and you catch it as an AbortError. No external libraries needed.

When a user navigates away from a page, any in-flight requests for that page are now pointless. If they resolve and try to update state, you get errors about updating unmounted components (very common in React) or stale data flashing on screen.

JavaScript — cleanup-navigation.js
// Vanilla JS — abort on page navigation
const controller = new AbortController()

async function loadPageData() {
  try {
    const data = await fetch("/api/page-data", {
      signal: controller.signal
    }).then(r => r.json())
    render(data)
  } catch (err) {
    if (err.name !== "AbortError") showError(err)
  }
}

// Cancel all pending requests when leaving
window.addEventListener("pagehide", () => controller.abort())

loadPageData()

Using AbortController for Timeouts

Modern browsers support AbortSignal.timeout(ms) — a one-liner that creates a signal that auto-aborts after a set time. No need to manage a controller manually for simple timeout cases.

JavaScript — timeout-signal.js
// AbortSignal.timeout() — built-in, no controller needed
try {
  const res = await fetch("/api/data", {
    signal: AbortSignal.timeout(5000)  // cancel if no response in 5s
  })
  const data = await res.json()
  console.log(data)
} catch (err) {
  if (err.name === "TimeoutError") {
    console.log("Request took too long")
  }
}

// Combine timeout AND manual abort — AbortSignal.any()
const controller = new AbortController()
const signal = AbortSignal.any([
  controller.signal,
  AbortSignal.timeout(5000)
])

// Now both a timeout OR manual abort will cancel the request
fetch("/api/data", { signal })
ℹ️ Browser Support

AbortSignal.timeout() is supported in all modern browsers and Node 18+. AbortSignal.any() is newer — check compatibility if supporting older browsers. The manual new AbortController() pattern works everywhere.

Preventing Double Submits

Another killer use case: users who click "Submit" twice. You need to either disable the button or cancel the first request — or both.

JavaScript — double-submit.js
let submitController = null

submitBtn.addEventListener("click", async () => {
  // If there's already a request in flight, cancel it
  if (submitController) submitController.abort()
  submitController = new AbortController()

  submitBtn.disabled = true
  submitBtn.textContent = "Submitting..."

  try {
    await fetch("/api/submit", {
      method: "POST",
      signal: submitController.signal,
      body: JSON.stringify(getFormData())
    })
    showSuccess()
  } catch (err) {
    if (err.name !== "AbortError") showError(err)
  } finally {
    submitBtn.disabled = false
    submitBtn.textContent = "Submit"
    submitController = null
  }
})
🧠 Mindset Shift

AbortController is about intentional cleanup. Most async code teaches you what to do when a request finishes. AbortController teaches you what to do when a request shouldn't finish — which is a completely different and equally important skill. Knowing when to cancel is part of building resilient UIs.
