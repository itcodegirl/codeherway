Pattern 1: Polling

Sometimes an API doesn't push updates to you — you have to go ask it repeatedly. This is polling, and it's older than most of us and still completely relevant.

Think: checking order status, monitoring a background job, watching for new messages in a system that doesn't have websockets. You ping the server every N seconds until something changes.

Short Polling
Ask every N seconds, no matter what
setInterval → fetch → update UI
Long Polling
Ask, server holds until answer, repeat
fetch → wait → fetch again on resolve
Adaptive Polling
Speed up when active, slow down when idle
Dynamic interval based on activity
Polling with Backoff
Back off on errors, retry exponentially
Error → double interval → try again

Basic Polling — The Wrong Way First

naive-polling.js
// ❌ The naive version — every junior dev writes this first setInterval(async () => { const data = await fetch('/api/order/status').then(r => r.json()); updateUI(data); }, 3000); // Problems: // 1. Keeps running forever, even after order is complete // 2. No error handling — one failure, next request still fires // 3. Requests can overlap if response takes longer than interval // 4. Can't be stopped // 5. Memory leak city if component unmounts

Polling Done Right

smart-polling.js
function createPoller({ url, interval = 3000, onData, onError, shouldStop }) { let timeoutId = null; let active = false; let currentInterval = interval; async function poll() { if (!active) return; try { const response = await fetch(url); if (!response.ok) throw new Error(`HTTP ${response.status}`); const data = await response.json(); // Reset interval on success currentInterval = interval; onData(data); // Check if we should stop if (shouldStop?.(data)) { stop(); return; } } catch (err) { // Exponential backoff on error, cap at 30s currentInterval = Math.min(currentInterval * 2, 30000); onError?.(err); } // Schedule next poll (never overlaps — sequential, not parallel) if (active) { timeoutId = setTimeout(poll, currentInterval); } } function start() { active = true; poll(); // Immediate first poll } function stop() { active = false; clearTimeout(timeoutId); } return { start, stop }; } // Usage — order status tracker const poller = createPoller({ url: `/api/orders/${orderId}/status`, interval: 3000, onData(data) { updateOrderStatus(data.status); }, shouldStop(data) { // Stop once order is terminal return ['delivered', 'cancelled', 'failed'].includes(data.status); }, onError(err) { console.warn('Poll failed, backing off:', err.message); } }); poller.start(); // Clean up when user leaves the page window.addEventListener('beforeunload', () => poller.stop());
✅ Key Design Decisions

Using setTimeout instead of setInterval means requests never overlap — the next one only fires after the previous one finishes. This is almost always what you want.

Pattern 2: Optimistic Updates

Here's the vibe: user clicks "like." Instead of making them stare at a spinner while you hit the API, you immediately update the UI as if it worked, then sync with the server in the background. If the server fails, you roll back.

Twitter does this. GitHub does this. Every app you love does this. It's the difference between "feels snappy" and "feels like a 2009 government website."

❌ Pessimistic (wait for server)
// User clicks like async function handleLike(postId) { // Show spinner setLoading(true); await fetch(`/api/posts/${postId}/like`, { method: 'POST' }); // Now update UI setLiked(true); setLoading(false); // 300ms of watching a spinner // for a button click. brutal. }
✅ Optimistic (update now, sync later)
async function handleLike(postId) { // Snapshot for rollback const prev = liked; // Update immediately ✨ setLiked(!liked); setCount(c => liked ? c - 1 : c + 1); try { await fetch(`/api/posts/${postId}/like`, { method: 'POST' }); } catch { // Roll back if it failed setLiked(prev); setCount(c => liked ? c + 1 : c - 1); showToast('Failed to save'); } }

Optimistic Updates for a List (Add + Delete)

optimistic-list.js
// Adding an item optimistically async function addTodo(text) { // Create a temporary item with a fake ID const tempId = `temp-${Date.now()}`; const optimisticItem = { id: tempId, text, status: 'saving...' }; // Add to UI immediately setTodos(prev => [...prev, optimisticItem]); try { const response = await fetch('/api/todos', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ text }) }); const realItem = await response.json(); // Replace temp item with real item from server setTodos(prev => prev.map(item => item.id === tempId ? realItem : item) ); } catch { // Remove the fake item on failure setTodos(prev => prev.filter(item => item.id !== tempId)); showToast('Could not add item. Try again.'); } } // Deleting an item optimistically async function deleteTodo(id) { const snapshot = [...todos]; // Save for rollback setTodos(prev => prev.filter(item => item.id !== id)); try { await fetch(`/api/todos/${id}`, { method: 'DELETE' }); } catch { setTodos(snapshot); // Restore the list showToast('Delete failed. Item restored.'); } }
⚠️ When NOT to Use Optimistic Updates

Financial transactions, irreversible actions, anything where "oops I rolled back" would be confusing or dangerous. If the cost of being wrong is high, show a spinner. Optimistic updates are for low-stakes, high-frequency interactions.

Pattern 3: Request Deduplication

Classic scenario: you have a user profile component that five different parts of your page all want to load. Without deduplication, you make five identical API calls. With deduplication, you make one and everyone shares the result.

This also applies to "the user triple-clicked the search button" situations. You want to make one request, not three.

In-Flight Request Cache

dedup-cache.js
// Map of in-flight requests by URL // key: URL string, value: the Promise itself const inFlight = new Map(); async function dedupedFetch(url, options = {}) { // Create a cache key from URL + method const key = `${options.method || 'GET'}:${url}`; // If there's already a request in flight for this URL, // return that same Promise — don't make a new request if (inFlight.has(key)) { console.log(`Deduped: ${key}`); return inFlight.get(key); } // Create the request Promise const promise = fetch(url, options) .then(r => r.json()) .finally(() => { // Remove from map when done (success or failure) inFlight.delete(key); }); inFlight.set(key, promise); return promise; } // These all fire at the same time — only one HTTP request is made const [a, b, c] = await Promise.all([ dedupedFetch('/api/user/profile'), dedupedFetch('/api/user/profile'), // deduped! dedupedFetch('/api/user/profile') // deduped! ]); // a === b === c (same object, same request)

Deduplication for Rapid Button Clicks

button-dedup.js
// A simple "only one active request at a time" guard function createSingleFlight(asyncFn) { let pending = null; return async function(...args) { if (pending) { console.log('Request already in flight, waiting...'); return pending; } pending = asyncFn(...args).finally(() => { pending = null; }); return pending; }; } // Wrap any async function const safeSave = createSingleFlight(async (data) => { const response = await fetch('/api/save', { method: 'POST', body: JSON.stringify(data) }); return response.json(); }); // User spam-clicks save — only one request fires saveBtn.addEventListener('click', () => safeSave(formData));

Pattern 4: Loading Skeletons

Spinners say "something is happening, who knows what." Skeletons say "your content is coming and it looks like this." The second one feels dramatically better, even when the load time is identical.

This isn't just aesthetics — skeletons also reduce layout shift, which is a Core Web Vital that affects your SEO. So now you have an excuse to tell your PM it's a performance issue.

CSS Skeleton Component

skeleton.css
/* The shimmer animation */ @keyframes shimmer { 0% { background-position: -200% 0; } 100% { background-position: 200% 0; } } .skeleton { background: linear-gradient( 90deg, #1e1e2e 25%, #2a2a3e 50%, #1e1e2e 75% ); background-size: 200% 100%; animation: shimmer 1.5s infinite; border-radius: 4px; } /* Reusable shape utilities */ .skeleton-text { height: 16px; margin-bottom: 8px; } .skeleton-title { height: 28px; margin-bottom: 16px; } .skeleton-avatar { width: 48px; height: 48px; border-radius: 50%; } .skeleton-card { height: 200px; border-radius: 12px; }

Swapping Skeleton ↔ Real Content

skeleton-swap.js
async function loadUserCard(userId) { const container = document.getElementById('user-card'); // Show skeleton immediately container.innerHTML = renderSkeleton(); try { const user = await fetch(`/api/users/${userId}`).then(r => r.json()); // Swap in real content container.innerHTML = renderUser(user); } catch (err) { container.innerHTML = renderError('Could not load user'); } } function renderSkeleton() { return ` <div class="user-card"> <div class="skeleton skeleton-avatar"></div> <div class="card-body"> <div class="skeleton skeleton-title" style="width:60%"></div> <div class="skeleton skeleton-text" style="width:90%"></div> <div class="skeleton skeleton-text" style="width:70%"></div> </div> </div> `; } function renderUser(user) { return ` <div class="user-card"> <img src="${user.avatar}" class="avatar" alt="${user.name}"> <div class="card-body"> <h3>${user.name}</h3> <p>${user.bio}</p> <span class="badge">${user.role}</span> </div> </div> `; }
✅ Skeleton Best Practices

Match the shape of your real content as closely as possible. A skeleton that's the wrong dimensions makes the layout jump when the real content loads — which defeats the entire purpose. Measure your real card, copy those dimensions.

💜 Dev_Fession

I spent two hours debugging why my poller was making 47 requests per second before I realized I'd called setInterval inside an event listener that fired on every keypress. The server logs looked like a DDoS. The server was not happy. I was not happy. Nobody was happy.

⚡ Quick Recap

Polling: setTimeout > setInterval, add backoff, always have a stop condition.
Optimistic updates: Snapshot → update → sync → rollback on error.
Deduplication: Cache the Promise itself, not just the result.
Skeletons: Match the shape of real content, swap on resolve or error.
