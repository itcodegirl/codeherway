You've Used async/await. But Do You Know Why It Works?

In the last series you learned to write async/await and fetch data from an API. You know the syntax. But there's a deeper question most tutorials skip: how does JavaScript actually do this? It runs one thing at a time — so how does it fetch data from the internet without freezing the entire page?

The answer is the event loop. Understanding it doesn't just satisfy curiosity — it directly explains why async code behaves the way it does, why certain bugs happen, and how to reason about the order your code runs in. This is the mental model that makes everything else in this series make sense.

🧱

JavaScript Is Single-Threaded

⏸ Pause & Process

The call stack is the most important piece to understand first. JavaScript is single-threaded — it can only run one thing at a time. The call stack tracks what's currently running. Each function call gets pushed onto the stack; when it returns, it gets popped off. The event loop's job is to keep the call stack fed without ever blocking it.

JavaScript has one call stack — one thread of execution. It can only do one thing at a time. This is not a weakness. It's a deliberate design decision that eliminates an entire class of concurrency bugs that plague multi-threaded languages.

The call stack is exactly what it sounds like: a stack of function calls. When you call a function, it gets pushed onto the stack. When it returns, it gets popped off. JavaScript works through that stack top to bottom, one frame at a time.

JavaScript — call-stack.js
function third()  { console.log("I run last") }
function second() { third() }
function first()  { second() }

first()

// Call stack at deepest point:
// [ third ]   ← top (currently running)
// [ second ]
// [ first  ]
// [ main   ]  ← bottom (global execution context)
//
// Each function pops off as it returns, bottom to top
ℹ️ Good to Know

You've seen the call stack before — it's what a stack trace shows you when an error occurs. Every line in that error output is a frame that was on the stack at the moment the error was thrown. The call stack isn't an abstract concept. It's the thing showing up in your DevTools right now.

⏳

The Problem: What About Things That Take Time?

If JavaScript is single-threaded and can only do one thing at a time, what happens when you fetch data that takes 500ms to arrive? Does the whole page freeze and wait?

The answer is no — and the reason is that the browser does the waiting, not JavaScript. When you call fetch(), JavaScript hands the request off to a Web API running in the browser. The browser handles the network request in the background. JavaScript immediately moves on to the next line of code. When the browser is done, it puts the callback on a queue. The event loop picks it up when the call stack is empty.

That's the whole system. Four pieces working together:

CALL STACK
currently running
function frames
one at a time
WEB APIs
fetch()
setTimeout()
DOM events
TASK QUEUE
waiting callbacks
ready to run
when stack is empty

The event loop is the traffic controller. It has one job: check if the call stack is empty. If it is, take the next callback from the queue and push it onto the stack. Repeat forever. That's it. That's the whole event loop.

🔬

Watching It Happen — setTimeout

⏸ Pause & Process

The microtask queue (where Promises live) and the task queue (where setTimeout lives) have a strict priority: microtasks always run before the next task. This is why Promise.resolve().then() runs before setTimeout(fn, 0) — even though zero milliseconds sounds instant. Once you internalize queue priority, async code ordering becomes predictable.

setTimeout is the clearest way to see the event loop in action. Watch what order these log in and think about why.

JavaScript — settimeout-order.js
console.log("1 — start")

setTimeout(() => {
  console.log("3 — setTimeout callback")
}, 0)  // 0ms delay — but still async!

console.log("2 — end")

// Output:
// 1 — start
// 2 — end
// 3 — setTimeout callback
//
// Even with 0ms delay, the callback goes to the Web API,
// then the queue, then runs AFTER the current stack clears.
// "0ms" means "as soon as possible" — not "right now".
🧠 Mindset Shift

setTimeout(fn, 0) doesn't mean "run immediately." It means "run after the current call stack finishes." This is one of the most misunderstood behaviors in JavaScript. The delay is a minimum wait time, not a guaranteed one. If the stack is busy for 2 seconds, your 0ms timeout still waits those 2 seconds.

⚡

The Microtask Queue — Promises Go Here

Here's where it gets more nuanced — and where async/await behavior starts to make complete sense. There are actually two queues, not one.

The task queue (also called the macrotask queue) handles things like: setTimeout, setInterval, and DOM events. The microtask queue handles: Promise callbacks (.then(), .catch()) and queueMicrotask().

The critical rule: microtasks always run before the next task. When the call stack empties, the event loop drains the entire microtask queue before picking up even one task from the task queue.

JavaScript — microtask-queue.js
console.log("1 — sync start")

setTimeout(() => console.log("4 — setTimeout (task queue)"), 0)

Promise.resolve().then(() => console.log("3 — Promise.then (microtask queue)"))

console.log("2 — sync end")

// Output:
// 1 — sync start
// 2 — sync end
// 3 — Promise.then (microtask queue)  ← runs before setTimeout!
// 4 — setTimeout (task queue)
//
// Microtasks always drain before the next task runs.
// This is why Promise callbacks feel "faster" than setTimeout.
⚡ Why This Matters for async/await

When you await something, JavaScript pauses that async function and puts the rest of it into the microtask queue. When the awaited Promise resolves, the continuation is pulled from the microtask queue — before any setTimeout callbacks. This is why async/await code resumes quickly and predictably. It's not magic. It's the microtask queue doing its job.

🗺️

The Full Picture — Step by Step

Let's trace exactly what happens when you run a real async function — one fetch call — through this entire system:

STEP 01
async function is called
It gets pushed onto the call stack. Synchronous code inside it runs normally.
STEP 02
await fetch() is hit
JavaScript hands the network request to the browser's Web API. The async function is paused. Its continuation is saved. The call stack moves on.
STEP 03
The rest of the synchronous code runs
Everything after the async function call continues. The browser fetches in the background. Nothing freezes.
STEP 04
The network response arrives
The browser resolves the Promise. The continuation of the async function is placed in the microtask queue.
STEP 05
The call stack empties
The event loop sees an empty stack. It checks the microtask queue first — finds the async function continuation.
STEP 06
The async function resumes
It picks up exactly where it left off — after the await. The response data is now available. The rest of the function runs.
JavaScript — event-loop-fetch.js
console.log("A — before async call")

async function loadData() {
  console.log("B — inside async, before await")

  const res = await fetch("https://jsonplaceholder.typicode.com/todos/1")

  // Everything below here is a microtask continuation
  // It doesn't run until the fetch resolves AND the stack is clear
  console.log("D — inside async, after await")
}

loadData()

console.log("C — after async call (sync continues)")

// Output:
// A — before async call
// B — inside async, before await
// C — after async call (sync continues)
// D — inside async, after await  ← arrives when fetch resolves
💡 Pro Tip

Run this exact code in your browser console right now. Watch the order. Then try adding another console.log after loadData() and before D — confirm that synchronous code always runs before the async continuation. This is the behavior that surprises every developer the first time they see it.

🚫

Blocking the Event Loop — What Not to Do

Because JavaScript is single-threaded, anything that occupies the call stack for too long blocks everything else. No events fire. No callbacks run. No UI updates. The page appears frozen.

JavaScript — blocking.js
// ❌ This blocks the event loop — don't do this
function blockFor2Seconds() {
  const start = Date.now()
  while (Date.now() - start < 2000) {}  // busy-wait loop
  console.log("Done blocking")
}

setTimeout(() => console.log("timeout fired"), 100)
blockFor2Seconds()

// "timeout fired" won't log until AFTER blockFor2Seconds completes
// because the while loop keeps the call stack occupied
// Output after ~2 seconds: "Done blocking"
// Then: "timeout fired"
//
// ✅ Expensive work should be broken into chunks, run async,
// or offloaded to a Web Worker so the event loop stays free
🧠 Mindset Shift

This is why you never see professional code doing heavy computation directly in JavaScript. Long-running synchronous operations starve the event loop. Best practice: break heavy work into async chunks, use setTimeout to yield control back between chunks, or use Web Workers for truly CPU-intensive tasks.
