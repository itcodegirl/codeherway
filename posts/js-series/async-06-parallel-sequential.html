The Most Expensive Mistake in Async Code

You can write perfectly correct async/await code that still makes your app feel slow. Not because of bugs — because of how you're ordering your awaits. This is the single most common performance mistake in JavaScript codebases, and it's invisible unless you know to look for it.

The issue is the difference between running Promises sequentially — one after another, each waiting for the last — and running them in parallel — all at once, waiting for all to finish. The result is identical. The time taken is dramatically different.

⏱️

Sequential vs Parallel — The Visual

⏸ Pause & Process

The sequential trap looks like correct code. Each await works. Each request succeeds. But each one waits for the previous to finish before starting — so if you have three requests that each take 1 second, sequential execution takes 3 seconds. Parallel execution takes 1 second. For a loading screen that difference is everything.

❌ Sequential (await one by one)
Request 1 — 300ms
Request 2 — 300ms
Request 3 — 300ms
Total: ~900ms (sum of all)
✅ Parallel (Promise.all)
Req 1
Req 2
Req 3
Total: ~300ms (slowest only)
JavaScript — sequential-vs-parallel.js
// ❌ Sequential — each await blocks the next from starting
async function loadSequential() {
  const user  = await fetchUser()   // waits 300ms
  const posts  = await fetchPosts()  // waits ANOTHER 300ms
  const photos = await fetchPhotos() // waits ANOTHER 300ms
  // total: ~900ms — and none of these depended on each other!
  return { user, posts, photos }
}

// ✅ Parallel — all three start at the same time
async function loadParallel() {
  const [user, posts, photos] = await Promise.all([
    fetchUser(),
    fetchPosts(),
    fetchPhotos()
  ])
  // total: ~300ms — same result, 3x faster
  return { user, posts, photos }
}

// When IS sequential correct?
// When each request DEPENDS on the result of the previous one
async function loadDependentData() {
  const user  = await fetchUser()           // need user first
  const posts = await fetchPosts(user.id)  // needs user.id — sequential is correct
  return { user, posts }
}
⚡ The Rule

If your async operations don't depend on each other's results, they should run in parallel. If they do depend on each other — where step 2 needs the result of step 1 — they must be sequential. Most of the time, you can run more in parallel than you think.

🔁

await in Loops — The Silent Performance Killer

⏸ Pause & Process

Controlled concurrency is the nuanced middle ground that comes up in real projects. You have 50 items to process but you don't want to fire 50 requests simultaneously — rate limits, server load, and memory all push back. Process them in batches of 5 using Promise.all() on slices. It's not complex code, but it shows genuine understanding of async patterns.

One of the most common places developers accidentally serialize async operations is inside loops. Using await inside a for loop makes each iteration wait for the previous one to complete — even when they don't need to.

JavaScript — async-in-loops.js
const ids = [1, 2, 3, 4, 5]

// ❌ Sequential loop — one request at a time, 5 trips total
async function fetchAllSequential() {
  const results = []
  for (const id of ids) {
    const res = await fetch(`/api/user/${id}`)  // each waits for prev
    results.push(await res.json())
  }
  return results
}

// ✅ Parallel loop — all 5 start at once
async function fetchAllParallel() {
  const promises = ids.map(id =>
    fetch(`/api/user/${id}`).then(r => r.json())
  )
  return Promise.all(promises)  // wait for all at once
}

// ❌ async forEach — looks async, isn't awaited by the loop
async function brokenForEach() {
  ids.forEach(async (id) => {
    await fetch(`/api/user/${id}`)
    // forEach returns void — it doesn't await your async callback
    // The function returns before any fetches complete
  })
  // You can't know when this is done — the Promises are just floating
}

// ✅ When you want sequential-but-controlled — for...of with await
async function sequential() {
  for (const id of ids) {
    await processUser(id)  // intentional sequencing
    // good when order matters or server rate-limits you
  }
}
⚠️ Never Use async forEach

Array.forEach() does not await async callbacks. It fires them all and moves on — you get no way to know when they finish, and errors get swallowed. Use for...of for sequential async iteration. Use .map() + Promise.all() for parallel. Never forEach with async.

🎛️

Controlled Concurrency — When Full Parallel Is Too Much

Running 1000 requests at once would overwhelm a server (and likely get you rate-limited). Sometimes you need a middle ground: run N at a time, rolling as each completes.

JavaScript — batch-parallel.js
// Process items in batches of N simultaneously
async function processInBatches(items, batchSize, processItem) {
  const results = []

  for (let i = 0; i < items.length; i += batchSize) {
    const batch = items.slice(i, i + batchSize)
    const batchResults = await Promise.all(batch.map(processItem))
    results.push(...batchResults)
  }

  return results
}

// Process 100 users, 5 at a time
const users = await processInBatches(
  userIds,
  5,
  id => fetch(`/api/user/${id}`).then(r => r.json())
)
💡 Pro Tip

The batch pattern is useful for more than just rate-limiting. It's also how you handle large datasets without memory issues — you process a manageable chunk at a time instead of loading everything into memory simultaneously.

⚡ DEV_FESSION

I built a data migration script that processed 500 records by awaiting each one inside a loop. It took 8 minutes to run. My senior dev looked at it for about 10 seconds, pointed at the await inside the for loop, and said "there's your problem." I switched to batches of 20 using Promise.all(). It ran in 23 seconds. I genuinely did not believe it was the same script. That was the day I started actually thinking about async execution strategy instead of just making code that worked.
