You Know GET. Now Learn the Rest.

Every example you've seen so far has been a basic GET request — the default when you call fetch(url) with no other arguments. But real APIs require you to POST data, include auth headers, send JSON, and handle HTTP status codes properly. This post covers the full fetch API — everything you need to interact with real-world APIs professionally.

📋

HTTP Methods

⏸ Pause & Process

HTTP methods are a communication convention. GET means 'give me data.' POST means 'create something new.' PUT/PATCH means 'update something.' DELETE means 'remove it.' When you're building a UI that talks to an API, knowing which method to use isn't optional — it's the contract between your code and the server.

MethodPurposeHas Body?
GETRead data — the defaultNo
POSTCreate new dataYes — sends the new data
PUTReplace existing data entirelyYes — the full replacement
PATCHUpdate part of existing dataYes — only the changed fields
DELETERemove dataSometimes — usually not
⚠️ The Error You Will See: CORS

CORS (Cross-Origin Resource Sharing) is the #1 error beginners hit with fetch. It looks like this in the console:

Access to fetch at 'https://api.example.com' from origin 'http://localhost' has been blocked by CORS policy.

Why it happens: Browsers block cross-origin requests unless the server explicitly allows them. This is a server-side security setting — you can't fix it from your JavaScript. What to do: Use an API that supports CORS (most public APIs do), use a CORS proxy for testing, or build a backend route that makes the request for you. If you're getting CORS errors with your own server, add the correct Access-Control-Allow-Origin response header.

The Full fetch() Options Object

JavaScript — fetch-options.js
// fetch() takes a URL and an optional options object
const response = await fetch("/api/posts", {
  method: "POST",          // GET, POST, PUT, PATCH, DELETE

  headers: {
    "Content-Type": "application/json",  // tells server we're sending JSON
    "Authorization": "Bearer my-token-here",
    "Accept": "application/json"
  },

  body: JSON.stringify({   // body must be a string — JSON.stringify() it
    title: "My New Post",
    body:  "Content here",
    userId: 1
  })
})

// GET requests CANNOT have a body — the browser ignores it
// POST/PUT/PATCH requests need both Content-Type header AND body

All Four HTTP Methods in Practice

⏸ Pause & Process

Status codes tell the whole story in two numbers. 200 = OK. 201 = Created (after a successful POST). 400 = your request was wrong. 401 = not authenticated. 403 = authenticated but not allowed. 404 = not found. 500 = server error. Reading the status code in the Network tab should be your first debugging step, every time.

JavaScript — crud-operations.js
const BASE = "https://jsonplaceholder.typicode.com"

// GET — fetch a post
async function getPost(id) {
  const res = await fetch(`${BASE}/posts/${id}`)
  if (!res.ok) throw new Error(res.status)
  return res.json()
}

// POST — create a new post
async function createPost(data) {
  const res = await fetch(`${BASE}/posts`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(data)
  })
  if (!res.ok) throw new Error(res.status)
  return res.json()  // returns the created resource with its new id
}

// PATCH — update only specific fields
async function updatePost(id, changes) {
  const res = await fetch(`${BASE}/posts/${id}`, {
    method: "PATCH",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(changes)
  })
  if (!res.ok) throw new Error(res.status)
  return res.json()
}

// DELETE — remove a post
async function deletePost(id) {
  const res = await fetch(`${BASE}/posts/${id}`, { method: "DELETE" })
  if (!res.ok) throw new Error(res.status)
  return true  // 204 No Content — nothing to parse
}

HTTP Status Codes You Must Know

Status codes tell you what happened on the server. response.ok is true for codes 200–299. Everything else is a failure — but you often want to know which kind of failure to give the user the right message.

JavaScript — status-codes.js
async function fetchWithStatusHandling(url) {
  const res = await fetch(url)

  switch (res.status) {
    case 200: return res.json()                          // OK
    case 201: return res.json()                          // Created
    case 204: return null                                 // No Content — don't call .json()
    case 400: throw new Error("Bad request — check your data")
    case 401: throw new Error("Unauthorized — log in first")
    case 403: throw new Error("Forbidden — you don't have access")
    case 404: throw new Error("Not found")
    case 429: throw new Error("Rate limited — slow down")
    case 500: throw new Error("Server error — not your fault")
    default:  throw new Error(`Unexpected status: ${res.status}`)
  }
}

Building a Reusable Fetch Wrapper

In real projects you'll make many fetch calls. Writing out all the headers and error handling every time is verbose and inconsistent. A fetch wrapper centralizes that logic so every call in your app behaves the same way.

JavaScript — api-client.js
// A reusable API client — use this as a starting point for any project
const api = {
  baseUrl: "https://jsonplaceholder.typicode.com",
  token: null,

  getHeaders() {
    const headers = { "Content-Type": "application/json" }
    if (this.token) headers["Authorization"] = `Bearer ${this.token}`
    return headers
  },

  async request(path, options = {}) {
    const res = await fetch(`${this.baseUrl}${path}`, {
      ...{headers: this.getHeaders()},
      ...options
    })
    if (res.status === 204) return null
    if (!res.ok) throw new Error(`${res.status} ${res.statusText}`)
    return res.json()
  },

  get(path)          { return this.request(path) },
  post(path, body)  { return this.request(path, { method: "POST",   body: JSON.stringify(body) }) },
  patch(path, body) { return this.request(path, { method: "PATCH",  body: JSON.stringify(body) }) },
  del(path)          { return this.request(path, { method: "DELETE"                          }) }
}

// Clean usage — no more boilerplate everywhere
const post    = await api.get("/posts/1")
const created = await api.post("/posts", { title: "New", body: "Content", userId: 1 })
const updated = await api.patch("/posts/1", { title: "Updated" })
await api.del("/posts/1")
💡 Pro Tip

This fetch wrapper pattern — sometimes called an API client or service layer — is how professional codebases work. You define it once, the whole app imports it. When the API changes its base URL or auth scheme, you change it in one place. Put this in your portfolio projects now. Interviewers will notice.

⚡ DEV_FESSION

I sent my first POST request and spent 20 minutes wondering why the server kept rejecting my data. I was sending a perfectly valid JavaScript object. What I didn't know: the body had to be a string. JSON.stringify() converts an object to a JSON string. Without it, you're literally sending [object Object] to the server. The server has no idea what to do with that. Neither did I, for about 20 minutes.
