Async JavaScript

The internet doesn't respond instantly. async/await is how JavaScript waits — without freezing.

⏱ 12 min read 🎯 Intermediate 🛠 Mini Build Included

JavaScript Has to Wait Sometimes

So far every line of code you've written runs instantly, in order, top to bottom. But real apps need to wait — wait for a server to respond, wait for a file to load, wait for a user to do something. That's asynchronous JavaScript, and it changes how you think about code entirely.

⏱️

The Problem: JavaScript Is Single-Threaded

⏸ Pause & Process

Promises are just a formalized way of saying 'I don't have this value yet, but I will.' They're not magic — they're objects with three states: pending, fulfilled, rejected. The reason async/await was invented is that Promise chains with .then().then().then() got hard to read. await just pauses execution until the Promise resolves, so you can write async code that reads top-to-bottom like normal code.

ℹ️ Why This Matters

JavaScript runs one thing at a time on a single thread. If it waited for every network request to finish before running anything else, your page would freeze every time you fetched data. Async lets JavaScript kick off a long operation and continue running other code, then come back and handle the result when it's ready.

🤝

Promises: A Contract for Future Values

JavaScript — promises.js
// A Promise represents a value that will arrive sometime in the future // It starts "pending" and eventually "resolves" (success) or "rejects" (fail) const myPromise = new Promise((resolve, reject) => { const success = true if (success) { resolve("Data loaded!") } else { reject(new Error("Something failed")) } }) myPromise .then(data => console.log(data)) // runs on success: "Data loaded!" .catch(error => console.error(error)) // runs on failure .finally(() => console.log("Done either way")) // Promise.all — run multiple async operations IN PARALLEL // Returns when ALL resolve, rejects if ANY fail const [users, posts] = await Promise.all([ fetch("/api/users").then(r => r.json()), fetch("/api/posts").then(r => r.json()), ]) // Both fetches run at the same time — much faster than sequential awaits
✨

async/await: Promises That Read Like Normal Code

⏸ Pause & Process

The most important thing to know about fetch(): it only rejects on network failure. A 404, a 500, a server error — none of those cause fetch() to throw. You have to check response.ok yourself. This is the number one beginner mistake with fetch, and it leads to bugs where errors are silently swallowed.

JavaScript — async-await.js
// async makes a function return a Promise // await pauses execution until the Promise resolves async function getUser() { try { const response = await fetch("https://jsonplaceholder.typicode.com/users/1") // ⚠️ CRITICAL: fetch ONLY rejects on network failure (offline, DNS, CORS) // 404, 500, and other HTTP errors do NOT cause fetch to reject // You MUST check response.ok to catch HTTP-level errors if (!response.ok) { throw new Error(`HTTP error: ${response.status}`) } const user = await response.json() console.log(user.name) } catch (error) { console.error("Fetch failed:", error.message) } finally { // runs whether fetch succeeded or failed hideLoadingSpinner() } }
⚠️ The #1 fetch Mistake

fetch does not throw on HTTP errors. A 404 "Not Found" or 500 "Server Error" response will not cause fetch to reject — it "succeeds" from JavaScript's perspective because the network request completed. Only a network failure (no internet, DNS error, CORS block) rejects a fetch promise. That's why you must always check response.ok. Skipping this check means your code silently proceeds with broken/empty data.

ℹ️ Quick Primer: JSON

When you fetch data from an API, it comes back as JSON — JavaScript Object Notation. It looks like a JavaScript object but it's actually just text. Two methods you'll use constantly:

response.json() — parses the JSON text into a real JavaScript object. This is what you call on every fetch response.

JSON.stringify(obj) — converts a JS object back to JSON text. Used when you want to save data to localStorage or send data in a POST request body.

A Complete Fetch Workflow

JavaScript — fetch-complete.js
async function loadPosts() { const container = document.querySelector("#posts") try { // Show loading state container.innerHTML = "<p>Loading...</p>" const response = await fetch("https://jsonplaceholder.typicode.com/posts?_limit=5") if (!response.ok) throw new Error(`HTTP error: ${response.status}`) const posts = await response.json() // Render posts into the DOM container.innerHTML = posts.map(post => ` <article> <h3>${post.title}</h3> <p>${post.body}</p> </article> `).join("") } catch (error) { container.innerHTML = `<p class="error">Failed to load: ${error.message}</p>` } finally { // Runs whether fetch succeeded or failed // Great place to hide loading spinners or re-enable buttons loadButton.disabled = false loadButton.textContent = "Load Posts" } }

No one told me fetch doesn't throw on 404. I thought it did. I thought HTTP errors were errors. They are. Just not to fetch. Always check response.ok. Always.
