JavaScript Scope, Hoisting & Closures

Understanding where your variables live — and why some mysteriously disappear — fixes an entire category of confusing bugs.

Why Do My Variables Disappear?

If you've ever gotten a ReferenceError: x is not defined and had no idea why — this post is for you. The answer is almost always scope: the rules that determine where a variable can be seen and used in your code.

🌍

The Three Scopes

⏸ Pause & Process

The Temporal Dead Zone is a fancy name for a simple idea: let and const declarations are hoisted (JavaScript knows they exist), but you can't access them before the line where they're declared. Try it and you'll get a ReferenceError. The TDZ is the gap between 'JavaScript knows the variable exists' and 'JavaScript will let you use it.'

JavaScript — scope.js
// 1. GLOBAL SCOPE — declared outside everything, accessible anywhere const siteName = "CodeHerWay" function greet() { // 2. FUNCTION SCOPE — only accessible inside this function const message = "Hello!" console.log(siteName) // ✅ can access global console.log(message) // ✅ can access own scope } // console.log(message) ← ❌ ReferenceError: message not defined outside if (true) { // 3. BLOCK SCOPE — only accessible inside this { } block (let/const) const blockVar = "I'm block scoped" console.log(blockVar) // ✅ works inside the block } // console.log(blockVar) ← ❌ ReferenceError outside the block
💀

Why var Is a Trap

❌ var — leaks out of blocks
if (true) {{
  var x = "leaked!"
}}
// var escapes the block
console.log(x)  // "leaked!" 😬
✅ let/const — stays in block
if (true) {{
  let x = "contained"
}}
// let stays in the block
console.log(x)  // ReferenceError ✅
🪄

Hoisting: JavaScript's Sneaky Move

⏸ Pause & Process

Closures are one of those concepts that sounds complicated until you realize you've been using them all along. Every time you write a function inside another function, the inner function 'closes over' the outer variables — it keeps access to them even after the outer function has finished running. That's all a closure is. Useful for things like keeping a counter's value private.

JavaScript — hoisting.js
// Function DECLARATIONS are fully hoisted — call before defining: works console.log(greet()) // ✅ "Hello!" — works before definition function greet() { return "Hello!" } // var is hoisted but value is undefined until the assignment line console.log(name) // undefined (not an error — just wrong) var name = "Jenna" console.log(name) // "Jenna" // let/const — TEMPORAL DEAD ZONE (TDZ) // They ARE hoisted, but NOT initialized // Accessing them before declaration throws ReferenceError // console.log(city) ← ❌ ReferenceError: Cannot access 'city' before initialization const city = "Chicago"
ℹ️ What Is the Temporal Dead Zone?

When JavaScript hoists a let or const declaration, the variable exists in memory but has no value — it's in a Temporal Dead Zone (TDZ) from the start of the block until the declaration line is reached. Accessing it during this window throws a ReferenceError. You'll see "TDZ" mentioned in documentation and error messages — now you know what it means.

🔮

Closure: Scope's Secret Superpower

A closure happens when a function remembers variables from its outer scope — even after that outer function has finished running. You use closures constantly without realizing it: every event handler you write that references an outer variable is a closure.

JavaScript — closure.js
// The inner function "closes over" the outer variable function makeCounter() { let count = 0 // private — can't be accessed from outside return function() { count++ return count } } const counter = makeCounter() console.log(counter()) // 1 console.log(counter()) // 2 console.log(counter()) // 3 // count is "remembered" even though makeCounter() finished running // You use closures constantly in event handlers: function makeClickHandler(message) { return () => console.log(message) // "remembers" message } btn.addEventListener("click", makeClickHandler("Hello!"))
🧠 Mindset Shift

Always declare variables before you use them. const and let give you a clear error if you get this wrong. var silently gives you undefined and lets you keep going — hiding the bug. That's exactly why we don't use it.

⚡ DEV_FESSION

I had a for loop using var i that was somehow affecting a variable called i elsewhere in my code. I spent two hours debugging it. Turns out var doesn't respect the loop's block — my loop's i was leaking into the outer function. Switched to let i. Fixed immediately. That was the last time I ever used var.
