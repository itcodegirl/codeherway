JavaScript Promises Explained

What a Promise actually is, how it's created, the three states it can be in, and how to handle what comes back.

A Promise Is a Contract

In Post 02 you saw what callback hell looked like and why the community wanted something better. Promises were the answer — introduced natively in ES6 (2015) and now a fundamental part of the language. But before you can use them fluently, you need to understand what they actually are, not just what they look like.

A Promise is an object that represents the eventual result of an asynchronous operation. Think of it like a restaurant receipt after ordering. The food isn't there yet — but you have a promise that it will arrive. You can decide in advance what you'll do when it shows up, and what you'll do if the kitchen says they can't make it.

🔄

The Three States

⏸ Pause & Process

The three Promise states are worth memorizing: pending (waiting), fulfilled (succeeded), rejected (failed). A Promise can only move in one direction — once it settles (fulfills or rejects), it stays there forever. That immutability is what makes Promises safe to pass around and chain.

Every Promise exists in exactly one of three states at any given moment. Once it moves out of pending, it never changes again — a settled Promise is permanent.

⏳
Pending
The async operation is still in progress. No result yet. This is the starting state of every Promise.
✅
Fulfilled
The operation completed successfully. The Promise has a resolved value you can work with.
❌
Rejected
The operation failed. The Promise has a reason — an error — explaining what went wrong.
⚡ Fulfilled vs Resolved

You'll hear "resolved" used two ways. Sometimes it means fulfilled (succeeded). But technically, a Promise can "resolve" to another Promise — meaning it's not settled yet. The spec term for success is fulfilled. In casual use, "resolved" usually means fulfilled. Don't let it trip you up — just know the distinction exists.

🏗️

Creating a Promise

You create a Promise with new Promise(), passing it an executor function that receives two arguments: resolve and reject. Call resolve(value) when the operation succeeds. Call reject(error) when it fails.

JavaScript — creating-promises.js
// Basic Promise structure
const myPromise = new Promise((resolve, reject) => {
  // Do async work here
  const success = true

  if (success) {
    resolve("It worked!")   // fulfilled — passes value to .then()
  } else {
    reject(new Error("It failed"))  // rejected — passes error to .catch()
  }
})

// Simulating async with setTimeout
const waitOneSecond = new Promise((resolve) => {
  setTimeout(() => {
    resolve("One second passed!")
  }, 1000)
})

// Real-world pattern — wrapping an old callback API in a Promise
function readFilePromise(path) {
  return new Promise((resolve, reject) => {
    fs.readFile(path, "utf8", (err, data) => {
      if (err) reject(err)
      else resolve(data)
    })
  })
}
ℹ️ Good to Know

In practice you rarely create Promises with new Promise() yourself — most async APIs already return Promises (fetch(), for example). But knowing how to construct one is essential for wrapping older callback-style APIs and for writing your own async utilities.

📬

.then(), .catch(), .finally()

⏸ Pause & Process

.finally() is the most underused Promise method. It runs whether the Promise succeeded or failed — perfect for cleanup: hiding a loading spinner, re-enabling a button, closing a connection. You don't need it for every Promise, but when you need it, nothing else does the job as cleanly.

Once you have a Promise, you handle its result with three methods. Each one takes a callback and returns a new Promise — which is what makes chaining possible (more on that in Post 04).

JavaScript — handling-promises.js
const promise = new Promise((resolve, reject) => {
  setTimeout(() => resolve("data arrived"), 500)
})

// .then() — runs when the Promise is fulfilled
promise.then((value) => {
  console.log("Success:", value)  // "Success: data arrived"
})

// .catch() — runs when the Promise is rejected
promise.catch((error) => {
  console.log("Error:", error.message)
})

// .finally() — runs regardless of success or failure
// Perfect for cleanup: hiding loaders, re-enabling buttons, etc.
promise.finally(() => {
  console.log("Always runs — clean up here")
})

// All three together — the standard pattern
promise
  .then(value  => console.log("Got:", value))
  .catch(error  => console.log("Failed:", error.message))
  .finally(()    => console.log("Done either way"))
⚡

Promise.resolve() and Promise.reject()

These static methods let you create instantly settled Promises — useful for testing, default values, and forcing async interfaces to accept sync values.

JavaScript — static-methods.js
// Promise.resolve() — creates an already-fulfilled Promise
Promise.resolve("instant value")
  .then(v => console.log(v))  // "instant value"

// Promise.reject() — creates an already-rejected Promise
Promise.reject(new Error("instant failure"))
  .catch(e => console.log(e.message))  // "instant failure"

// Real use case — returning cached data as a Promise
// so the calling code always deals with a Promise interface
function getUser(id) {
  const cached = cache.get(id)
  if (cached) return Promise.resolve(cached)  // sync data, async interface
  return fetch(`/api/users/${id}`).then(r => r.json())
}
⚠️

Common Promise Mistakes

JavaScript — promise-mistakes.js
// ❌ MISTAKE 1: Forgetting to return in .then()
// The chain breaks — next .then() gets undefined
fetch("/api/user")
  .then(response => {
    response.json()  // ← missing return!
  })
  .then(data => console.log(data))  // data is undefined

// ✅ Correct — return the inner Promise
fetch("/api/user")
  .then(response => response.json())  // implicit return ✓
  .then(data => console.log(data))

// ❌ MISTAKE 2: Nesting .then() instead of chaining
// You've recreated callback hell with Promises
fetch("/api/user").then(res => {
  res.json().then(user => {     // ← nesting, not chaining
    console.log(user)
  })
})

// ❌ MISTAKE 3: No .catch() handler
// Rejected Promises become unhandled rejections — bad in production
fetch("/api/user")
  .then(res => res.json())
  // no .catch() — if this fails, the error disappears silently
⚠️ Unhandled Promise Rejections

In modern Node.js, an unhandled Promise rejection crashes your process. In browsers, it logs a console warning — but the error is silently swallowed from the user's perspective. Always add a .catch(). No exceptions. This is one of the first things a code reviewer will flag.

⚡ DEV_FESSION

I spent an embarrassing amount of time debugging a feature where data was always undefined after a fetch. I checked the API. I checked the network tab. I checked the response. Everything looked fine. Turns out I forgot to return inside my .then() and the entire chain was getting undefined passed through it. The missing return is the silent killer of Promise chains. I've never forgotten it since. Burn it into your brain: if you're inside a .then() and calling something that returns a Promise, return it.
