What "State" Actually Means

State is any data that, when it changes, should cause the UI to update. The current user. Whether a modal is open. A list of loaded posts. The selected city in a weather app. These aren't static — they change over time in response to user actions and async data.

Frameworks like React and Vue manage state for you with clever abstractions. But knowing how to manage state in vanilla JS is what makes you understand what those frameworks are actually doing — and what makes you dangerous in interviews. This post builds a mini reactive store from scratch.

The Problem With Variables Everywhere

⏸ Pause & Process

'State' is just a fancy word for 'data that changes over time and should cause the UI to update when it does.' The current user. Whether a modal is open. A list of search results. The loading status. Once you see your app through the lens of state, you'll understand why React, Vue, and Svelte exist — and why you should understand this pattern before reaching for them.

JavaScript — spaghetti-state.js
// The "scatter variables everywhere" approach — works until it doesn't
let currentUser  = null
let isLoading    = false
let posts        = []
let selectedCity = "chicago"
let error        = null

// When one changes, you manually update the UI everywhere it's used
async function loadPosts() {
  isLoading = true
  renderLoading()     // remember to update the UI

  try {
    posts = await fetchPosts()
    isLoading = false
    renderPosts()      // remember to update the UI
    renderStatus()     // remember to update the UI
  } catch (err) {
    error     = err
    isLoading = false
    renderError()      // remember to update the UI
    renderStatus()     // remember to update AGAIN
  }
}
// The problem: state is scattered, UI updates are manual and easy to miss

A Reactive Mini Store

The solution: a single source of truth. One object holds all state. When state changes, subscribers are notified automatically. This is the same model React's useState, Vue's reactive(), and Redux all use.

JavaScript — mini-store.js
// A minimal reactive store — ~25 lines, no dependencies
function createStore(initialState) {
  let state       = { ...initialState }
  let subscribers = []

  return {
    // Read current state
    getState() {
      return { ...state }  // spread prevents accidental mutation
    },

    // Update state and notify subscribers
    setState(updates) {
      state = { ...state, ...updates }
      subscribers.forEach(fn => fn({ ...state }))
    },

    // Subscribe to state changes
    subscribe(fn) {
      subscribers.push(fn)
      return () => {
        subscribers = subscribers.filter(s => s !== fn)  // unsubscribe
      }
    }
  }
}

// Create a store for our weather app
const store = createStore({
  city:      "chicago",
  weather:   null,
  loading:   false,
  error:     null
})

// Subscribe — render is called automatically on any state change
store.subscribe(state => render(state))

// Async action — updates state, subscribers do the rendering
async function fetchWeather(city) {
  store.setState({ loading: true, error: null })
  try {
    const weather = await getWeather(city)
    store.setState({ weather, loading: false })
  } catch (err) {
    store.setState({ error: err.message, loading: false })
  }
}
⚡ This Is What React Does

React's useState is this store pattern, baked in. When you call setState(), React re-renders your component — which is React calling its subscribers. When you use useEffect, you're subscribing to state changes. Understanding this model makes learning React dramatically easier.

Connecting State to the DOM

⏸ Pause & Process

The subscriber pattern (pub/sub, observer pattern) is one of the most widely used patterns in UI development. Your store calls all its subscribers whenever state changes. Each subscriber re-renders the part of the UI it owns. This is exactly what React's useState and Redux do. Building it yourself first means you're never confused by what a framework is actually doing.

JavaScript — dom-rendering.js
// One render function driven entirely by state
function render(state) {
  const app = document.querySelector("#app")

  if (state.loading) {
    app.innerHTML = "<p>Loading weather...</p>"
    return
  }

  if (state.error) {
    app.innerHTML = `
      <div class="error">
        <p>${state.error}</p>
        <button id="retry">Try again</button>
      </div>
    `
    document.querySelector("#retry")
      .addEventListener("click", () => fetchWeather(state.city))
    return
  }

  if (state.weather) {
    const { temperature_2m, wind_speed_10m } = state.weather.current
    app.innerHTML = `
      <h2>${state.city.toUpperCase()}</h2>
      <p>${temperature_2m}°F — Wind: ${wind_speed_10m} mph</p>
    `
  }
}

// City selector triggers a state update — render handles the rest
document.querySelector("#city-select").addEventListener("change", e => {
  store.setState({ city: e.target.value })
  fetchWeather(e.target.value)
})

Adding localStorage Persistence to the Store

JavaScript — persistent-store.js
// A store that automatically persists to localStorage
function createPersistentStore(key, initialState) {
  // Restore from localStorage on init, fall back to initialState
  const saved   = storage.get(key)
  const store   = createStore({ ...initialState, ...saved })

  // Persist to localStorage on every state change
  store.subscribe(state => {
    storage.set(key, {
      city: state.city  // only persist what makes sense to persist
    })
  })

  return store
}

// Now city preference survives page refresh
const store = createPersistentStore("weather-app", {
  city: "chicago", weather: null, loading: false, error: null
})
🧠 Mindset Shift

This is the pattern your portfolio projects should follow. Instead of scattered variables and manual DOM updates, one store, one render function, automatic persistence. Your code becomes easier to reason about, easier to debug, and easier to explain in interviews — because you can point to exactly where all your state lives.

⚡ DEV_FESSION

I once built a weather app with 14 different global variables. They were all over the place — some in functions, some at the top of the file, some inside event handlers. It worked. It also took me an hour to find a bug where isLoading wasn't being reset in the error path because I'd updated the variable but forgotten to call the render function. When I rebuilt it with a store pattern, that bug became impossible — setState always notified subscribers, there was no way to change state without the UI updating. Impossible bugs are better than fixed bugs.
