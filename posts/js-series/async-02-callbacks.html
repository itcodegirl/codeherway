JavaScript Callbacks Explained

Callbacks were JavaScript's first answer to async. Understanding why they broke down is the whole reason Promises exist.

⏱ 11 min read 🎯 Intermediate 🛠 Mini Build Included

To Understand Where We Are, You Need to Know Where We've Been

Every tutorial teaches you Promises and async/await and says "this is how you handle async code." Very few explain why those things were invented — what problem they were solving, and what that problem looked like before they existed. That context is what makes Promises feel like a relief instead of just another syntax to memorize.

The answer starts with callbacks — the original async pattern in JavaScript. You actually already know what a callback is. You've been writing them since Post 10 of the JS Click series.

🧱

What a Callback Actually Is

⏸ Pause & Process

Callbacks aren't bad. The problem is nesting. A single callback is clean and fine. Two nested callbacks are still readable. Three or four nested callbacks — each with their own error handling — is where 'callback hell' earns its name. The code starts looking like a pyramid of doom, and tracing what happens when is genuinely difficult.

← First introduced in JS Click — Post 10: Events: Making Things Interactive

A callback is just a function passed as an argument to another function, to be called later. You've done this constantly with event listeners and array methods — you just didn't always call it by name.

JavaScript — callbacks-you-know.js
// You've been writing callbacks all along

// addEventListener — the second argument IS a callback
btn.addEventListener("click", function() {
  console.log("clicked!")  // runs later, when the click happens
})

// setTimeout — callback runs after the delay
setTimeout(function() {
  console.log("later!")
}, 1000)

// Array methods — callback runs once per item
[1, 2, 3].forEach(function(n) {
  console.log(n)
})

// The pattern: pass a function, it gets called for you later
// That's all a callback is. You already know this.

For simple cases, callbacks work perfectly. The problem shows up when you need to do several async things in sequence — one after another, each depending on the result of the last. That's where callbacks started to break down.

🕳️

Callback Hell

Before fetch() existed, JavaScript used a function called XMLHttpRequest — one of the most notoriously verbose APIs ever written. But the problem wasn't the specific API. It was what happened when you needed to make multiple async calls where each one depended on the result of the previous one.

JavaScript — callback-hell.js
// Real code from pre-Promise JavaScript.
// Get a user → get their posts → get comments on first post

getUser(1, function(err, user) {
  if (err) {
    console.log("Error getting user:", err)
    return
  }

  getPosts(user.id, function(err, posts) {
    if (err) {
      console.log("Error getting posts:", err)
      return
    }

    getComments(posts[0].id, function(err, comments) {
      if (err) {
        console.log("Error getting comments:", err)
        return
      }

      getAuthor(comments[0].userId, function(err, author) {
        if (err) {
          console.log("Error getting author:", err)
          return
        }

        // Finally — the actual thing we wanted to do
        console.log("Author:", author.name)
      })
    })
  })
})

This is callback hell. Also called the pyramid of doom — named for the triangular shape the code makes as it indents deeper and deeper. Each nested level is another async operation waiting on the one above it.

⚠️ PYRAMID OF DOOM — DEPTH METER
Level 1
getUser()
Level 2
getPosts(user.id)
Level 3
getComments(posts[0].id)
Level 4
getAuthor(comments[0].userId)
Level 5+
any further operations...

The Four Problems With Callbacks

⏸ Pause & Process

Callbacks are still the right tool in many situations: event listeners, array methods (.map(), .filter()), and Node.js file system operations. The problem was using them for sequential async operations — fetching data, then processing it, then saving it. That's exactly what Promises and async/await were designed for.

The pyramid shape was the visual symptom. The actual problems were deeper:

JavaScript — callback-problems.js
// PROBLEM 1: Error handling is manual and repetitive
// You have to check for errors at every single level.
// Miss one and errors silently disappear.
doSomething(function(err, result) {
  if (err) { /* handle it */ return }  // repeated everywhere
  doNextThing(result, function(err, result2) {
    if (err) { /* handle it again */ return }  // and again
  })
})

// PROBLEM 2: Inversion of control
// You're handing YOUR function to THEIR code.
// You trust them to call it once, at the right time.
// What if they call it twice? Never? With wrong args?
thirdPartyAnalytics.track("purchase", function() {
  chargeCard()  // if this gets called twice... 😬
})

// PROBLEM 3: No way to run things in parallel easily
// Need results from A and B before doing C?
// You'd have to manually track completion state with flags.
let userDone = false, postsDone = false
getUser(function(err, u)   { userDone  = true; if (postsDone) finish() })
getPosts(function(err, p) { postsDone = true; if (userDone)  finish() })

// PROBLEM 4: Can't use try/catch
// Errors thrown inside callbacks escape try/catch blocks
try {
  setTimeout(function() {
    throw new Error("This error is uncatchable!")
  }, 100)
} catch (e) {
  console.log("This never runs")  // the error already escaped
}
⚡ The Inversion of Control Problem

This one is subtle but important. When you pass a callback to third-party code, you're surrendering control of when and how your function runs. You're trusting that code completely — that it will call your function exactly once, with the right arguments, at the right time. That's a lot of trust to place in code you didn't write and can't inspect. Promises were designed specifically to take that control back.

✅

The Same Code With Promises

You don't need to understand Promises deeply yet — that's Posts 03 and 04. But seeing the before and after is the whole point of this post. Here's that same four-level async chain, written with Promises:

❌ Callbacks — the old way
getUser(1, function(err, user) {
  if (err) return
  getPosts(user.id,
    function(err, posts) {
      if (err) return
      getComments(
        posts[0].id,
        function(err, comments) {
          // deeper and deeper...
        }
      )
    }
  )
})
✅ Promises — the modern way
getUser(1)
  .then(user =>
    getPosts(user.id)
  )
  .then(posts =>
    getComments(posts[0].id)
  )
  .then(comments => {
    // flat. readable. clean.
  })
  .catch(err => {
    // ONE handler for all errors
  })

Same logic. Flat structure. One error handler at the bottom instead of one at every level. And then async/await made it even cleaner — but we'll get there. The point right now is understanding why the community moved away from callbacks. It wasn't arbitrary. It was a direct response to real, painful problems.

🧠 Mindset Shift

Callbacks aren't inherently bad. Simple, single-level callbacks — like event listeners and array methods — are perfectly fine and you'll keep writing them forever. The problem was specifically nested async callbacks where each level depended on the previous one. That's the pattern Promises were invented to replace. Know the difference.

📖

When Callbacks Are Still the Right Tool

Now that you understand the problem, here's an important nuance: callbacks didn't disappear. They're still everywhere — and in many cases they're exactly the right tool.

JavaScript — callbacks-still-valid.js
// ✅ Event listeners — callbacks are perfect here
btn.addEventListener("click", () => handleClick())

// ✅ Array methods — callbacks are perfect here
const doubled = [1, 2, 3].map(n => n * 2)

// ✅ Simple one-time async — still fine
setTimeout(() => hideToast(), 3000)

// ✅ Node.js file system (traditional style)
fs.readFile("data.json", "utf8", (err, data) => {
  if (err) throw err
  console.log(data)
})

// ❌ Multiple dependent async operations — use Promises/async/await
getUser(function(err, user) {
  getPosts(user.id, function(err, posts) {
    // stop here, reach for Promises instead
  })
})
💡 Pro Tip

The rule of thumb: if your callback is going to nest, stop and reach for a Promise instead. One level of callback is fine. Two or more levels that each depend on the previous result is callback hell territory — and that's exactly what Promises and async/await were invented to flatten.

⚡ DEV_FESSION

The first real codebase I read as a junior dev was written entirely in callback style — this was older Node.js code. I spent two days trying to understand a single feature because the logic was spread across six levels of nesting and the error handling was inconsistent across all of them. I thought I was too dumb to understand it. Turns out the code was genuinely hard to follow — not because I was new, but because deeply nested callbacks are hard for everyone. That was the day I really understood why the community moved on. It wasn't about aesthetics. It was about survival.

    Now You Know Why Promises Were Invented

    Callbacks weren't a mistake — they were the right tool for the time, and they're still the right tool for simple cases. But nested async callbacks introduced real problems: unreadable structure, repeated error handling, inversion of control, and broken try/catch. Promises were the direct, deliberate solution to every one of those problems. Next post: how Promises actually work, from the inside out.
