JavaScript Debounce vs Throttle

Two tools that control how often a function fires — and the difference between them is the difference between a smooth app and a crushed server.

The Problem: Too Many Function Calls

Some events fire constantly. A user typing fires input on every keystroke. A user scrolling fires scroll tens of times per second. A window resize fires resize hundreds of times while dragging. If you're making fetch calls, updating the DOM, or doing any expensive work in those handlers — you have a performance problem.

Debounce and throttle are the two standard solutions. They both limit how often a function runs, but they do it in fundamentally different ways — and choosing the wrong one makes your UI feel broken.

⚖️

Debounce vs Throttle — The Difference

⏸ Pause & Process

The difference between debounce and throttle trips people up. Here's the clearest mental model: debounce says 'wait until you stop doing that, then fire once.' Throttle says 'you can fire, but not more often than once every X milliseconds.' Search boxes use debounce (wait until typing stops). Scroll progress bars use throttle (update every 100ms, not every frame).

⏳ Debounce
Wait until they stop. Delays execution until the event hasn't fired for N milliseconds. Perfect for search inputs — you want to search after the user finishes typing, not on every keystroke.
⚡ Throttle
Limit to once per N ms. Guarantees the function runs at most once per interval, no matter how many events fire. Perfect for scroll — you want regular updates, just not 60 per second.
⚡ The One-Line Memory Aid

Debounce: "Call me when you're done." — Throttle: "Call me, but not more than once a second." If you can only remember one rule: search inputs get debounce, scroll handlers get throttle.

🏗️

Building Debounce From Scratch

JavaScript — debounce.js
// Debounce — delays calling fn until N ms after the last call
function debounce(fn, delay) {
  let timer

  return function(...args) {
    clearTimeout(timer)  // cancel the previous scheduled call
    timer = setTimeout(() => fn.apply(this, args), delay)
  }
}

// Usage — search fires 300ms after the user stops typing
const debouncedSearch = debounce(async (query) => {
  const results = await searchAPI(query)
  renderResults(results)
}, 300)

document.querySelector("#search").addEventListener("input", e => {
  debouncedSearch(e.target.value)
})

// What happens when you type "hello" fast:
// h    → timer set for 300ms
// he   → timer cancelled, new timer set for 300ms
// hel  → timer cancelled, new timer set for 300ms
// hell → timer cancelled, new timer set for 300ms
// hello → timer cancelled, new timer set for 300ms
// (user stops typing)
// 300ms later → search("hello") fires ONCE ✓

Building Throttle From Scratch

⏸ Pause & Process

Clearing the timer is the crucial line in every debounce implementation. Without clearTimeout(timer), every keystroke creates a new timer and none of them cancel the old one — you've just added a delay, not a debounce. The whole point is that each new call resets the clock. One final call after the user stops — that's debounce.

JavaScript — throttle.js
// Throttle — guarantees fn runs at most once per interval
function throttle(fn, interval) {
  let lastCall = 0

  return function(...args) {
    const now = Date.now()

    if (now - lastCall >= interval) {
      lastCall = now
      return fn.apply(this, args)
    }
    // else: discard the call — too soon
  }
}

// Usage — scroll position logged at most every 200ms
const throttledScroll = throttle(() => {
  const scrollY = window.scrollY
  updateProgressBar(scrollY)
}, 200)

window.addEventListener("scroll", throttledScroll)

// What happens when you scroll for 1 second:
// Scroll event fires ~60 times (60fps)
// Throttle at 200ms = fires at: 0ms, 200ms, 400ms, 600ms, 800ms, 1000ms
// Result: 6 calls instead of 60 ← 10x fewer DOM updates

Real-World Use Cases

JavaScript — use-cases.js
// DEBOUNCE use cases:

// 1. Search input — wait until user stops typing
const search = debounce(query => fetchResults(query), 300)

// 2. Form auto-save — don't save every keystroke
const autoSave = debounce(data => saveToServer(data), 1000)

// 3. Window resize — recalculate layout after done resizing
const onResize = debounce(() => recalcLayout(), 150)
window.addEventListener("resize", onResize)

// THROTTLE use cases:

// 1. Scroll position — update header/progress bar smoothly
const onScroll = throttle(() => updateHeader(), 100)
window.addEventListener("scroll", onScroll)

// 2. Mouse move — only update every 50ms during drag
const onMouseMove = throttle(e => updateDragPosition(e), 50)

// 3. Game loop / canvas — cap update rate
const gameUpdate = throttle(() => renderFrame(), 16)  // ~60fps cap
💡 Lodash Has These Too

Lodash's _.debounce() and _.throttle() are battle-tested with additional options like leading/trailing edge control. In a real project, reach for lodash if it's already in your dependencies. In interviews, you'll be asked to implement them from scratch — which is why you just did.

Combining With AbortController

Debounce reduces how often you fetch. AbortController cancels stale fetches. Together they're the complete solution for search inputs — far fewer requests, and the ones that do fire can still be cancelled if a new one starts.

JavaScript — debounce-abort.js
// The complete search pattern: debounce + AbortController
let controller

const search = debounce(async (query) => {
  if (controller) controller.abort()
  controller = new AbortController()

  try {
    const res = await fetch(`/api/search?q=${query}`, {
      signal: controller.signal
    })
    const data = await res.json()
    renderResults(data)
  } catch (err) {
    if (err.name !== "AbortError") showError(err)
  }
}, 300)

// Debounce: waits 300ms before even making the request
// AbortController: cancels it if the user types again in that 300ms
// Together: essentially 0 wasted requests
⚡ DEV_FESSION

I built a project dashboard that filtered a list of cards on every keypress in a search box. No debounce. It worked fine with 20 cards. Then the client added 2,000 cards and a CSS filter effect that recalculated on every input event. On slow machines, the page froze while typing. Users thought it was broken. Adding a 150ms debounce — one function, one line of change — made it smooth as glass. The client thought I'd done major performance work. I told them the truth, mostly. I left out the part where it took me 30 seconds.
