They return data in formats you have to read documentation to understand. This post is your field guide to navigating all of it.

🔑

Authentication Patterns

⏸ Pause & Process

API documentation is a skill in itself. Every API is different, but they all tell you the same things: the base URL, what parameters are accepted, what the response shape looks like, and what authentication is required. The fastest path through unfamiliar docs: find the authentication section first, then find the simplest endpoint (usually something like /users or /posts), and make one working request before building anything.

Most real APIs require you to prove who you are. There are a few common patterns:

JavaScript — auth-patterns.js
// PATTERN 1: API Key in header (most common for public APIs)
const res = await fetch("https://api.example.com/data", {
  headers: {
    "X-API-Key": "your-api-key-here"
  }
})

// PATTERN 2: Bearer token (OAuth, JWTs)
const res = await fetch("https://api.example.com/data", {
  headers: {
    "Authorization": `Bearer ${token}`
  }
})

// PATTERN 3: API key in query string (some older APIs)
const res = await fetch(
  `https://api.example.com/data?api_key=${API_KEY}`
)

// ⚠️ NEVER hardcode API keys in frontend JS
// They are visible to anyone who opens DevTools.
// For keys that must stay secret, use a backend proxy.
// For public read-only keys — fine to include, just don't commit them.
// Use environment variables: process.env.API_KEY or import.meta.env.VITE_KEY
⚠️ API Keys in Frontend Code

Any API key you put in your JavaScript is visible to users who open DevTools. For truly secret keys (Stripe secret key, private data), you need a backend. For public read-only keys (weather APIs, maps), it's common practice — just make sure the API has domain restrictions so others can't use your key.

⚠️ Never Put API Keys in Client-Side JavaScript

Anyone can open DevTools and see every variable, every fetch URL, every header. If your API key is in your JavaScript file, it's public. Full stop.

For portfolio projects, use a backend proxy, Netlify/Vercel serverless functions, or a .env file with a build tool. For learning, stick to APIs that don't require keys — like Open-Meteo (covered at the end of this post). Never commit secrets to GitHub.

Query Parameters Done Right

Query parameters are how you filter, sort, search, and paginate API responses. Don't build them with string concatenation — use URLSearchParams to handle encoding automatically.

JavaScript — query-params.js
// ❌ Manual string building — breaks with special characters
const url = `/api/search?q=${query}&page=${page}`
// If query = "C++ basics", url breaks: ?q=C++ basics

// ✅ URLSearchParams — handles encoding automatically
const params = new URLSearchParams({
  q:       "C++ basics",   // → q=C%2B%2B+basics
  page:    1,
  limit:   20,
  sort:    "date",
  order:   "desc"
})

const res = await fetch(`/api/search?${params}`)

// Utility — build URL with params cleanly
function buildURL(base, params) {
  const url = new URL(base)
  Object.entries(params).forEach(([key, val]) => {
    if (val !== null && val !== undefined) {
      url.searchParams.set(key, val)
    }
  })
  return url.toString()
}

const searchURL = buildURL("https://api.example.com/posts", {
  q: "async javascript", page: 2, tag: null  // null gets skipped
})

Pagination

⏸ Pause & Process

Rate limiting is real and it will happen to you. The polite engineering response is not to just hammer the endpoint and catch 429 errors — it's to build backoff logic (covered in Post 08) and to cache responses you already have (covered in Post 11). A request that doesn't get made is faster and cheaper than one that does.

Real APIs don't return all their data at once — that would be thousands of records per request. They paginate. There are two common approaches: page-based (page 1, page 2...) and cursor-based (give me the next N items after this ID).

JavaScript — pagination.js
// PAGE-BASED PAGINATION
// API returns: { data: [...], total: 500, page: 1, perPage: 20 }
async function fetchPage(page = 1) {
  const res = await fetch(`/api/posts?page=${page}&limit=20`)
  return res.json()
}

// Fetch all pages (careful — could be a LOT of requests)
async function fetchAll() {
  let page = 1, allData = []
  while (true) {
    const { data, total, perPage } = await fetchPage(page)
    allData.push(...data)
    if (allData.length >= total) break
    page++
  }
  return allData
}

// CURSOR-BASED PAGINATION (GitHub, Twitter/X style)
// API returns: { data: [...], nextCursor: "abc123" | null }
async function fetchWithCursor(cursor = null) {
  const params = new URLSearchParams({ limit: 20 })
  if (cursor) params.set("cursor", cursor)
  const res = await fetch(`/api/posts?${params}`)
  return res.json()
}

Rate Limits — Respecting the Server

Most APIs cap how many requests you can make per minute or hour. Exceed the limit and you get a 429 Too Many Requests response. The fix is to check the rate limit headers the API returns and back off when needed.

JavaScript — rate-limits.js
// Most APIs return rate limit info in headers
const res = await fetch("/api/data")

// Common header names (vary by API)
const remaining = res.headers.get("X-RateLimit-Remaining")
const resetAt   = res.headers.get("X-RateLimit-Reset")  // Unix timestamp
const limit     = res.headers.get("X-RateLimit-Limit")

console.log(`${remaining} of ${limit} requests remaining`)

// Handle 429 — wait and retry
async function fetchRespectingRateLimit(url) {
  const res = await fetch(url)

  if (res.status === 429) {
    const retryAfter = res.headers.get("Retry-After") || 60
    console.log(`Rate limited. Waiting ${retryAfter}s...`)
    await new Promise(r => setTimeout(r, retryAfter * 1000))
    return fetchRespectingRateLimit(url)  // retry after waiting
  }

  return res.json()
}

Reading API Documentation

Every API has docs. Knowing what to look for is half the battle — you don't need to read every page, just find the things that matter.

Authentication
How do I prove I'm me? API key? OAuth? Bearer token? Where does it go — header or query param?
Base URL
What's the root URL for all requests? Is there a versioned path like /v2/? Is there a sandbox vs production URL?
Rate Limits
How many requests per minute/hour? What headers tell me my current usage? What happens when I exceed it?
Response Shape
What does the JSON actually look like? Is data nested? Is there a wrapper object like { data: [...] }? What are the field names?
Error Format
What does a failed response look like? Is there an error message in the body? What status codes does it use?
Pagination
Page-based or cursor-based? What params control it? What does the response tell me about whether there's a next page?
💡 Pro Tip: Read the Error Docs First

Most devs skip to the endpoint docs and only read the error section when something goes wrong. Read the error format first. Knowing what a failed response looks like before you write a single line of code will save you hours of debugging.

A Free API Worth Knowing: Open-Meteo

We'll use the Open-Meteo weather API throughout this series — and build a full capstone with it in Post 18. It's free, requires no API key, and returns real weather data.

JavaScript — open-meteo.js
// Open-Meteo — free weather API, no key needed
// Docs: open-meteo.com/en/docs

async function getWeather(lat, lon) {
  const params = new URLSearchParams({
    latitude:           lat,
    longitude:          lon,
    current:            "temperature_2m,wind_speed_10m,weather_code",
    temperature_unit:   "fahrenheit",
    wind_speed_unit:    "mph"
  })

  const res = await fetch(
    `https://api.open-meteo.com/v1/forecast?${params}`
  )
  if (!res.ok) throw new Error(res.status)
  return res.json()
}

// Chicago, IL
const weather = await getWeather(41.85, -87.65)
console.log(weather.current.temperature_2m)  // temp in °F
console.log(weather.current.wind_speed_10m)  // wind in mph
