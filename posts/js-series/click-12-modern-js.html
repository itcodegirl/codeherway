Modern JavaScript (ES6+)

Template literals, destructuring, spread, optional chaining — the features that make modern JavaScript readable.

Write Less. Mean More.

Modern JavaScript (ES6 and beyond) introduced a set of features that made the language dramatically more readable and expressive. You'll see these in every codebase, every tutorial, and every job interview. This post covers the ones you'll actually use every single day.

📝

Template Literals

⏸ Pause & Process

Destructuring is the one feature in this post most worth slowing down for. It works on both arrays and objects, and it shows up everywhere: function parameters, API responses, React props, import statements. The core idea is always the same: instead of pulling values out one at a time, you pull them all out in one line by matching the structure of the source.

JavaScript — template-literals.js
const name = "Jenna" const role = "frontend dev" // Old way — messy string concatenation console.log("Hello, I'm " + name + " and I'm a " + role) // Template literal — use backticks + ${} expressions console.log(`Hello, I'm ${name} and I'm a ${role}`) // Expressions inside ${} — not just variables const score = 82 console.log(`Grade: ${score >= 90 ? "A" : "B or lower"}`) // Multi-line strings — no more hacks const card = ` <div class="card"> <h2>${name}</h2> <p>${role}</p> </div> `
📦

Destructuring

JavaScript — destructuring.js
// Object destructuring — pull out named properties const user = { name: "Jenna", age: 27, city: "Chicago" } // Long way const name = user.name const age = user.age // Destructuring — same result, one line const { name, age, city } = user console.log(name, age, city) // "Jenna" 27 "Chicago" // Rename while destructuring const { name: displayName } = user // Default value if property doesn't exist const { country = "US" } = user // Array destructuring — pull out by position const [first, second, , fourth] = ["a", "b", "c", "d"] console.log(first, second, fourth) // "a" "b" "d" (skipped "c") // In function parameters — super common with API responses function greetUser({ name, age }) { return `${name} is ${age}` } greetUser(user) // "Jenna is 27"
↔️

Spread & Rest

⏸ Pause & Process

Optional chaining (?.) and nullish coalescing (??) solve the same root problem: working with values that might not exist. ?. stops early and returns undefined instead of throwing. ?? provides a fallback only for null/undefined (unlike || which also replaces 0 and ''). Together they make 'maybe this value exists' code almost pleasant to write.

JavaScript — spread-rest.js
// Spread — expand an array or object into individual items const a = [1, 2, 3] const b = [4, 5, 6] const combined = [...a, ...b] // [1, 2, 3, 4, 5, 6] // Copy an array (not a reference) const copy = [...a] // Spread objects — merge or override properties const defaults = { theme: "dark", lang: "en" } const userPrefs = { lang: "fr" } const settings = { ...defaults, ...userPrefs } // { theme: "dark", lang: "fr" } — userPrefs.lang overrides defaults.lang // Rest — gather remaining items into an array const [head, ...tail] = [1, 2, 3, 4] console.log(head) // 1 console.log(tail) // [2, 3, 4] // Rest in function params — accept any number of args function sum(...nums) { return nums.reduce((a, b) => a + b, 0) } sum(1, 2, 3, 4) // 10
🔗

Optional Chaining & Nullish Coalescing

JavaScript — optional-chaining.js
const user = { name: "Jenna", address: { city: "Chicago" } } const incompleteUser = { name: "Maya" } // Without optional chaining — throws TypeError if address is undefined // console.log(incompleteUser.address.city) ← TypeError! // Optional chaining (?.) — returns undefined instead of throwing console.log(user?.address?.city) // "Chicago" console.log(incompleteUser?.address?.city) // undefined (no crash) // Also works for method calls — guards against calling undefined as a function user.getName?.() // undefined if getName doesn't exist, no TypeError // ?? (Nullish coalescing) — fallback for null/undefined ONLY const displayName = user.nickname ?? "Guest" // "Guest" (nickname is undefined) // ?? vs || — the key difference const count = 0 console.log(count || 10) // 10 — because 0 is falsy, || replaces it console.log(count ?? 10) // 0 — because 0 is NOT null/undefined // Assignment operators (ES2021) — assign only if the condition applies let name = null name ??= "Guest" // assign only if null/undefined → "Guest" name ||= "Fallback" // assign only if falsy name &&= "Updated" // assign only if currently truthy
🧠 ?? vs || — Don't Mix Them Up

|| replaces any falsy value — including 0, "", and false. ?? only replaces null and undefined. When providing fallbacks for missing data, ?? is usually what you want — || will silently replace legitimate zero values and empty strings.

⚡ DEV_FESSION

I used || as a fallback for months before discovering ??. My cart item count showed "Add some items" when the count was zero — because 0 || "Add some items" returns the fallback. 0 ?? "Add some items" returns 0. That one character fix saved me from a confusing bug in production.
