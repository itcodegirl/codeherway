1. HTML Entities — Special Characters

Some characters have special meaning in HTML. The < symbol starts a tag. The & symbol starts an entity. If you want to display these characters as text instead of having the browser interpret them as code, you need HTML entities.

HTML · Entities in Practice
<!-- Displaying HTML code as text --> <p>Use &lt;strong&gt; to make text bold.</p> <!-- Renders: Use <strong> to make text bold. --> <!-- Copyright footer --> <footer>&copy; 2026 CodeHerWay &mdash; All rights reserved.</footer> <!-- Non-breaking space to prevent awkward line breaks --> <p>Price: $9&nbsp;/month</p> <!-- "$9" and "/month" will never split across lines -->
💡 When You Need Entities

You must use entities for <, >, and & when they appear in text content — otherwise the browser interprets them as HTML. For other symbols like © and —, you can also type the actual Unicode character directly if your charset is UTF-8 (which it should be). But entities make your code more explicit and portable.

2. Data Attributes — Your Custom Data Layer

This is where HTML starts talking to JavaScript. Data attributes let you store custom information directly on any HTML element — information that JavaScript can read, filter, and act on. They all follow the same pattern: data-*.

HTML · Data Attributes
<!-- Storing metadata on elements --> <div class="course-card" data-course-id="html-101" data-difficulty="beginner" data-price="9"> <h3>HTML Fundamentals</h3> <p>Start your coding journey here.</p> </div> <!-- Button that knows what it controls --> <button data-action="toggle-menu" data-target="#main-nav"> Menu </button> <!-- Product with sortable data --> <li data-category="javascript" data-level="intermediate" data-release="2026-03"> JavaScript Deep Dive </li>

Data attributes are invisible to users but accessible to JavaScript via element.dataset. When you build filters, sort features, tabs, accordions, or any interactive component — data attributes are how the HTML tells JavaScript what to do.

JavaScript · Reading Data Attributes
// Access via dataset (camelCase conversion) const card = document.querySelector('.course-card'); card.dataset.courseId // "html-101" card.dataset.difficulty // "beginner" card.dataset.price // "9" // Note: data-course-id becomes dataset.courseId // Hyphens convert to camelCase automatically
ℹ️ Rules for Data Attributes

The name must start with data- followed by at least one character. Use only lowercase letters, numbers, hyphens, dots, colons, and underscores. The value is always a string — if you need a number, you'll parse it in JavaScript. And you can have as many data attributes on a single element as you need.

3. Microdata — Talking to Search Engines

Microdata is structured data you add to your HTML to help search engines understand what your content means — not just what it says. When Google shows star ratings, recipe times, event dates, or FAQ dropdowns in search results, that's microdata (or its siblings, JSON-LD and RDFa) at work.

HTML · Microdata Example
<!-- A course with structured data --> <div itemscope itemtype="https://schema.org/Course"> <h2 itemprop="name">HTML Fundamentals</h2> <p itemprop="description"> Learn HTML from scratch — the complete beginner course. </p> <span itemprop="provider">CodeHerWay</span> </div>

The itemscope attribute says "this element contains structured data." The itemtype points to a vocabulary (schema.org) that defines what kind of thing it is. And itemprop labels each piece of data within it.

🧠 You Don't Need to Master This Yet

Microdata is an introduction — you should know it exists and what it does. In practice, most developers use JSON-LD (a script-based format) instead of inline microdata because it's easier to maintain. But understanding the concept — that you can tell search engines "this is a course, this is its name, this is the provider" — is the important part.

4. The <template> Element — Hidden Until Needed

The <template> element holds HTML that the browser does not render until JavaScript explicitly clones and inserts it. It's perfect for components that need to be created dynamically — like cards that get added when data loads, or rows that appear when users click "Add Item."

HTML + JS · Template Element
<!-- HTML: Define the template (invisible on page) --> <template id="course-template"> <article class="course-card"> <h3 class="course-title"></h3> <p class="course-desc"></p> <span class="course-level"></span> </article> </template> // JavaScript: Clone and use it const template = document.getElementById('course-template'); const clone = template.content.cloneNode(true); clone.querySelector('.course-title').textContent = 'CSS Mastery'; clone.querySelector('.course-desc').textContent = 'Flexbox, Grid, and beyond.'; clone.querySelector('.course-level').textContent = 'Intermediate'; document.querySelector('.course-list').appendChild(clone);
💡 Why Not Just Use innerHTML?

You could build HTML strings in JavaScript and inject them with innerHTML. But templates are cleaner, safer (no XSS injection risk), and keep your HTML structure in the HTML file where it belongs — not buried inside JavaScript strings. When you move to frameworks like React, this same concept becomes "components."

5. DOM Structure Awareness

You've seen the DOM tree in DevTools. Now it's time to understand it conceptually — because this understanding is the bridge to JavaScript.

The DOM (Document Object Model) is the browser's live representation of your HTML. Every element becomes a node in a tree structure. Nodes have parent-child relationships, sibling relationships, and they can be selected, modified, created, and removed by JavaScript.

DOM Relationships
<!-- This HTML... --> <main> <!-- parent --> <h1>Title</h1> <!-- child, sibling of p --> <p>Paragraph</p> <!-- child, sibling of h1 --> </main> // ...creates this DOM tree: // // main (parent) // ├── h1 (firstChild) // │ └── "Title" (text node) // └── p (lastChild) // └── "Paragraph" (text node) // // main.children → [h1, p] // h1.parentElement → main // h1.nextElementSibling → p // p.previousElementSibling → h1

When you write document.querySelector('.card') in JavaScript, you're searching this tree. When you call element.appendChild(), you're adding a new node. Every JavaScript framework — React, Vue, Angular — ultimately manipulates this same DOM tree. Understanding its structure now means JavaScript will make sense immediately when you get there.

🧠 Think in Nodes

Start looking at your HTML not as text, but as a tree. Every element is a node with a parent, potential children, and potential siblings. This mental model is the foundation for CSS selectors, JavaScript DOM manipulation, and framework component trees. The HTML you write is the blueprint — the DOM is the building the browser constructs from it.

Build an HTML-only course catalog that's ready for JavaScript to bring it to life:

1
Create four course cards. Each card should have data-category (html, css, javascript, react), data-level (beginner, intermediate, advanced), and data-price attributes.
2

3
Use HTML entities to display code examples within the card descriptions — e.g., "Learn &lt;div&gt; and &lt;span&gt; elements."

5
Include a copyright footer using the &copy; entity, an &mdash; em dash, and a &nbsp; non-breaking space between "CodeHerWay" and the year.

The Bottom Line

These intermediate concepts are the bridge between "I can write HTML" and "I can build dynamic applications." Entities let you display special characters safely. Data attributes store custom information on elements for JavaScript to use. Templates hold reusable HTML that renders on demand. And DOM awareness gives you the mental model to understand how browsers — and JavaScript — actually work with your code.

You're not just writing markup anymore. You're designing a data structure that JavaScript, CSS, and search engines can all interact with. That's intermediate thinking.
