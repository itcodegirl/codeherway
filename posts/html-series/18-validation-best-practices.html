Validation & Best Practices:
Writing Code Like a Pro

Knowing HTML syntax isn't enough. Professionals write code that's clean, consistent, validated, and readable — code that other developers (including future you) can understand at a glance.

#HTML #BestPractices #CleanCode 10 min read

There's a difference between code that works and code that's good. Working code displays the right content in the browser. Good code does that and is clean, maintainable, accessible, validated, and readable by anyone who opens the file — including you six months from now.

This post covers the five practices that separate beginner HTML from professional HTML. None of them are complicated. All of them are habits. Start using them now and they'll become automatic.

The Five Best Practices

✅
W3C Validation
Run your code through the official validator to catch silent errors that browsers hide from you. Zero errors = solid foundation.
📐
Clean Indentation
Consistent spacing that reveals your document's structure at a glance. Two spaces or one tab — pick one, never mix.
📛
Consistent Naming
Lowercase, hyphenated file names and IDs. No spaces, no camelCase in HTML. One convention across the entire project.
🏷️
Meaningful Classes
Class names that describe what an element IS, not what it looks like. .card-title beats .big-red-text every time.
🏗️
Logical Hierarchy
Semantic elements in the right order. Headings that don't skip levels. A structure that reads like an outline, not a jumble.

1. W3C Validation — Your Code's Spellcheck

The W3C HTML Validator is the official tool for checking whether your HTML is valid. Think of it as a spellcheck for your code — it catches errors you didn't know were there.

Go to validator.w3.org, paste your code or URL, and hit Check. It returns a list of every error and warning in your document — unclosed tags, invalid attributes, incorrect nesting, missing required elements, and more.

Validator Workflow
// Step 1: Write your HTML // Step 2: Go to validator.w3.org // Step 3: Paste code or enter URL // Step 4: Fix errors from top to bottom // (one fix often resolves multiple errors) // Step 5: Re-validate until clean // Pro tip: Also available as VS Code extensions // Search "W3C Validation" or "HTMLHint" in extensions
💡 Fix Top-Down

When the validator returns 20 errors, don't panic. Fix the first error and re-validate. Often a single unclosed tag at the top of the document causes a cascade of errors below it. Fixing one error might eliminate ten others. Always work from the top of the error list down.

2. Clean Indentation — Structure You Can See

Indentation isn't about aesthetics — it's about communication. Proper indentation reveals the parent-child relationships in your HTML at a glance. When you see two elements at the same indent level, you know they're siblings. When one is indented deeper, you know it's a child.

HTML · Indentation
<!-- ❌ Inconsistent indentation --> <main> <section> <h2>About</h2> <p>Some text.</p> <p>More text.</p> </section> <section> <h2>Contact</h2> </section> </main> <!-- ✅ Consistent 2-space indentation --> <main> <section> <h2>About</h2> <p>Some text.</p> <p>More text.</p> </section> <section> <h2>Contact</h2> </section> </main>
ℹ️ Tabs vs Spaces

The great debate: 2 spaces, 4 spaces, or tabs? It doesn't matter which you pick. What matters is that you pick one and use it everywhere. Most web teams use 2 spaces. Set this in your editor's settings (VS Code: "Tab Size: 2" and "Insert Spaces: true") and never think about it again.

3. Consistent Naming — One Convention, Everywhere

File names, IDs, and classes should follow the same pattern across your entire project. The web standard is lowercase with hyphens (kebab-case). No spaces. No uppercase. No underscores (in HTML — CSS methodologies like BEM use them, which is fine).

Naming Conventions
// ❌ Inconsistent naming index.html AboutPage.html my contact.html <div class="MainHeader"> <div id="side_bar"> <div class="card wrapper"> <!-- two classes, not one --> // ✅ Consistent kebab-case index.html about.html contact.html <header class="main-header"> <aside id="side-bar"> <div class="card-wrapper">

Consistent naming prevents bugs (case-sensitive servers break About.html vs about.html), makes your code searchable, and shows other developers that you care about craft. It's a small thing that signals professionalism.

4. Meaningful Class Names — Describe the What, Not the How

This is where beginners and professionals diverge most visibly. Beginners name classes based on how things look. Professionals name classes based on what things are.

HTML · Class Names
<!-- ❌ Appearance-based names --> <div class="big-red-text">...</div> <div class="left-box">...</div> <div class="blue-border">...</div> <div class="box1">...</div> <!-- ✅ Purpose-based names --> <div class="error-message">...</div> <aside class="sidebar">...</aside> <div class="card-featured">...</div> <div class="testimonial">...</div>

Why? Because appearance changes. You might redesign and that "big red text" becomes small and blue. Now the class name is a lie. But an error-message is always an error message — regardless of how it's styled. Name things by their role, not their appearance.

🧠 The Newspaper Test

If someone read only your class names — without seeing the page — could they understand the structure? hero-section, course-card, pricing-table, testimonial-quote — these tell a story. box1, red-thing, container3 — these tell nothing. Aim for class names that describe the content's purpose.

5. Logical Hierarchy — HTML as an Outline

Your HTML document should read like a structured outline. Headings shouldn't skip levels. Semantic elements should wrap the right content. The order of elements should follow a logical flow that makes sense even without CSS.

HTML · Logical Structure
<!-- ❌ Skipped heading levels, illogical order --> <h1>My Portfolio</h1> <h4>About Me</h4> <!-- Skipped h2 and h3 --> <h2>Projects</h2> <h6>Project One</h6> <!-- Skipped h3, h4, h5 --> <!-- ✅ Sequential hierarchy --> <h1>My Portfolio</h1> <section> <h2>About Me</h2> <p>...</p> </section> <section> <h2>Projects</h2> <article> <h3>Project One</h3> <p>...</p> </article> <article> <h3>Project Two</h3> <p>...</p> </article> </section>

The rule is simple: h1 → h2 → h3 → h4. Never skip a level. Screen readers generate a navigable table of contents from your headings — if you skip from h1 to h4, there are gaps in that outline that confuse users.

💡 The Disable-CSS Test

Try disabling CSS in your browser (DevTools → Elements → uncheck all styles). Does your page still make sense? Can you read it top-to-bottom and understand the content hierarchy? If yes, your HTML structure is solid. If it's a jumbled mess without CSS, your HTML is relying too much on styling for structure.

// dev_fession

I used to name my CSS classes after how they looked: .big-bold, .red-box, .left-column. Then I redesigned my portfolio and moved the sidebar to the right side of the page. The class was still called .left-column. Rendering on the right.

A developer friend looked at my code and said, "Why is your right sidebar called left-column?" I had no good answer.

That's when I switched to naming things by what they are: .sidebar, .error-alert, .course-card. The names survived every redesign after that. 🏷️

The Pre-Deploy Checklist

Before you ship any page, run through this checklist. Print it out. Tape it to your monitor. Make it a habit until you don't need it anymore:

✅ HTML Quality Checklist
☐
Passes W3C Validator with zero errors and zero warnings
☐
DOCTYPE, charset, viewport, and title are all present
☐
Only one <h1> per page, headings don't skip levels
☐
Semantic elements used (header, nav, main, section, article, footer)
☐
Every <img> has a meaningful alt attribute
☐
Every form input has a <label> with matching for/id
☐
All links have descriptive text (not "click here")
☐
Indentation is consistent (2 spaces throughout)
☐
File names are lowercase with hyphens, no spaces
☐
Class names describe purpose, not appearance
☐
Page makes sense with CSS disabled
☐
Tested on mobile (DevTools device mode or real device)

🔨 Mini Build: Code Audit Your Portfolio

Take your portfolio (or any previous project from this series) and audit it against professional standards:

1
Run it through the W3C Validator. Fix every error and warning until you reach zero. Document how many errors you found.
2
Check your indentation. Is it consistent? If not, reformat the entire file with 2-space indentation. Configure your editor to do this automatically going forward.
3
Review every class name and ID. Replace any appearance-based names with purpose-based names. Update the corresponding CSS.
4
Check your heading hierarchy. Does your page have one h1? Do headings follow h1 → h2 → h3 order without skipping? Fix any gaps.
5
Disable CSS in DevTools and read your page. Does the content still make sense in order? If not, restructure your HTML until it reads like a proper document without any styling.

The Bottom Line

Professional HTML isn't about knowing more tags — it's about the habits you build around the tags you already know. Validate your code. Indent consistently. Name things by their purpose. Follow a logical heading hierarchy. And audit your work before you ship it.

These five practices are what interviewers notice when they look at your code. They're what teammates appreciate when they inherit your project. And they're what you'll appreciate when you come back to your own code six months later and can actually understand it.

Write HTML that works. Then make it HTML that's good.
